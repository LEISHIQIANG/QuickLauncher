## 1. 总体结论

项目已经过了“基础功能补齐”阶段，当前核心风险主要不是单点功能缺失，而是工程化收口不足：

- 测试数量充足，但反馈速度被 favicon/Qt/SVG 路径拖慢。
- 发布脚本能产出安装包，但发布洁净度、签名、手工安装矩阵、构建环境一致性还需要补强。
- 稳定性能力已经很多，例如配置恢复、插件隔离、诊断包、更新 session、安全模式，但文档和验收清单明显滞后。
- `优化计划书.md` 指出的底层质量问题仍有多项成立：Core 反向导入 UI、全局 service-locator 状态、配置迁移非原子、鼠标钩子桩接口、单例初始化无锁、静默异常治理不足。
- 新手引导和运行环境检测需要重新纳入稳定性范围：欢迎页内容明显过期，Python/Git/Git Bash 缺失虽然部分路径会报错，但诊断中心尚未形成统一环境检测。
- 代码复杂度集中在少数千行级模块，继续堆功能会增加回归成本。
- 类型检查和覆盖率还没有形成可执行的长期质量门禁。
- 已有治理工具（`error_handler.py`、`_clipboard_lock`、`ConfigDataStore`）存在但未被使用或绕过，形成"基建空转"。
- Qt 资源管理（QThread/QTimer 生命周期）存在系统性遗漏。

优先级建议：

- P0：发布前必须处理或明确验收的风险。
- P1：短期必须治理，否则会持续拖慢迭代或增加支持成本。
- P2：中期工程质量和维护性治理。
- P3：产品能力演进，不能压过稳定性和发布质量。

## 2. P0 发布前必须检查

- [ ] [P0][风险: 高] 建立 Windows 10/11 真实安装验收记录
  现状：自动产物校验已通过，但只能证明文件齐全、版本和哈希符合脚本规则，不能证明安装、升级、卸载、开机自启、中键钩子在真实系统中可用。
  优化方向：为 Windows 10、Windows 11 分别建立手工验收表，覆盖首次安装、覆盖安装、旧版本升级、卸载残留、普通用户启动、管理员启动、开机自启、UAC 弹窗、托盘退出。
  验收标准：每次发版都有一份安装验收记录，记录系统版本、安装路径、是否管理员、安装包哈希、关键操作结果和失败日志路径。

- [ ] [P0][风险: 高] 自动更新链路需要做端到端演练
  现状：更新检查、下载、哈希校验、session、安装前配置备份、首次启动确认都有代码和测试覆盖，但还缺一轮真实 GitHub Release 到安装器启动的完整演练记录。
  优化方向：用测试 release 或本地 mock release 流程验证 `check -> download -> sha256 -> backup -> installer -> first_start_confirmed -> diagnostics` 全链路。
  验收标准：更新 session 中能看到下载、安装、备份、首次启动确认状态；诊断包能导出 update session；失败时 UI 能给出可执行提示。

- [ ] [P0][风险: 中] 安装包未见代码签名流程
  现状：`.gitignore` 已忽略证书和密钥文件，但构建脚本中未发现 `signtool` 或签名步骤。公开分发时可能触发 SmartScreen、杀软拦截或用户信任问题。
  优化方向：补充可选签名阶段，支持未配置证书时跳过，配置证书时签名 exe 和 installer，并把签名状态写入 release manifest。
  验收标准：发布包可验证 Authenticode 签名；未签名构建在 release manifest 中明确标记 `signed=false`。

- [ ] [P0][风险: 中] 构建日志中的环境警告需要归档和处理
  现状：本次构建日志出现 `mt.exe not found; skipping manifest embedding`、Nuitka 对 PyQt5 的兼容警告、PyQt6/PySide6 冲突提示、TDM-GCC 架构不匹配候选被忽略。
  优化方向：把这些警告分为“可忽略”和“必须修复”。`mt.exe` 缺失需要确认是否影响打包 exe manifest、DPI、UAC 行为；PyQt6/PySide6 冲突需要保证发布环境不混装；编译器候选需要明确优先级。
  验收标准：构建日志中的已知 warning 都在文档中有解释；新增 warning 需要阻断发版或进入 release issue。

- [ ] [P0][风险: 中] 发布产物通过校验，但校验规则还不够严格
  现状：`check_release_artifacts.py` 已确认 exe、hooks.dll、assets、plugins、installer 存在并生成 sha256。但发布包中仍存在 `__pycache__` 和 `.pyc`，说明当前校验不会检查发布洁净度。
  优化方向：扩展 release gate，禁止 `__pycache__`、`.pyc`、临时上传配置、构建中间日志进入 `dist/QuickLauncher` 和安装包。
  验收标准：`scripts/check_release_artifacts.py` 能报告发布包内的缓存、临时文件、隐藏上传配置文件，并在 release 构建中失败退出。

## 4. P1 安全与权限边界

- [ ] [P1][风险: 高] 插件权限仍是提示性边界，不是强隔离
  现状：插件系统已有 manifest 校验、权限提示、隔离、错误日志、超时和 UI 状态；设置页也提示 in-process 兼容模式未强隔离。真实安全边界仍然是“只安装可信插件”。
  优化方向：把插件信任模型写入用户文档和插件开发文档首页；对高风险权限增加安装时确认；长期考虑进程级插件 runner 或受限 API broker。
  验收标准：安装高风险插件时用户必须确认；诊断包能列出高风险插件；文档明确“权限声明不是沙箱”。

- [ ] [P1][风险: 中] 命令执行安全需要持续保持回归场景
  现状：命令风险检测、变量替换、安全预处理、超时终止、输出截断已有测试覆盖。风险点在于 CMD、PowerShell、Python、Git Bash、动作链之间 quoting 和变量传递规则复杂。
  优化方向：建立命令安全回归矩阵，覆盖 `{{clipboard}}`、`{{input}}`、路径空格、引号嵌套、多行文本、危险 shell 元字符、管理员执行和动作链透传。
  验收标准：每次改动命令执行、变量替换、动作链前必须跑对应安全用例；新增运行时必须先补风险检测和文档。

- [ ] [P1][风险: 中] 运行时环境检测也应纳入安全边界
  现状：如果 PATH 中的 `python`、`git`、`bash` 被错误程序或恶意同名程序遮蔽，当前 `/path-audit` 可以发现部分 PATH 问题，但命令运行时选择和诊断中心没有统一展示“实际会执行哪个二进制”。
  优化方向：环境检测不仅检查“有没有”，还要展示解析到的完整路径、版本和风险提示。对 WindowsApps Python alias、非 Git for Windows 的 `bash.exe`、异常路径中的 `git.exe` 给出 warn。
  验收标准：诊断中心和命令测试结果能显示实际 launcher 路径；高风险路径有提示；用户能复制环境检测报告。

- [ ] [P1][风险: 中] 诊断脱敏已增强，但需要建立隐私验收样本
  现状：`core/diagnostics.py` 已有 token、api-key、password、Authorization、Cookie、用户目录等脱敏规则，也会输出 `redaction_report.json` 和 `events.jsonl`。缺口是没有独立的隐私样本清单覆盖更多真实日志形态。
  优化方向：增加诊断包隐私 golden tests，覆盖 URL query、HTTP header、命令行参数、JSON 字段、Windows 路径、用户名、机器名、插件错误 trace。
  验收标准：测试诊断包内不得出现样本 token、密码、用户名原文；redaction report 计数稳定。

- [ ] [P1][风险: 中] URL 图标获取安全需要保留为高优先级门禁
  现状：favicon 测试覆盖 data URL、manifest、inline SVG、私网阻断、loopback redirect、外部引用拒绝等安全场景。该模块又是慢测试热点，存在被误降级的风险。
  优化方向：拆分快慢测试时不能减少 SSRF、防外部引用、redirect 阻断、data URL 尺寸限制等安全断言。
  验收标准：慢测试拆分后，安全相关 favicon 测试仍在 CI 默认门禁或安全门禁中运行。

## 5. P1 测试体系与质量门禁

- [ ] [P1][风险: 高] 本地环境与 CI 依赖不一致
  现状：本地 `E:\Python312` 缺少 `pytest-cov/coverage`，运行 CI 覆盖率命令会报 `unrecognized arguments: --cov...`。CI 会安装 `requirements-dev.txt`，本地未安装。
  优化方向：增加 `scripts/setup_dev_env.bat` 或 README 中明确 `E:\Python312\python.exe -m pip install -r requirements.txt -r requirements-dev.txt`；也可以保留无虚拟环境策略，但必须让本地和 CI 命令一致。

- [ ] [P2][风险: 中] 单例初始化缺少统一线程安全模式

  - `HooksDLL.get_instance()` (`hooks_wrapper.py:39-43`)：无锁，check-then-act TOCTOU；DLL 加载失败时 `cls._instance.dll is None` 每次调用都重新构造，反复加载 DLL。

  优化方向：对跨线程可能访问的单例使用双检锁或模块级线程安全初始化；纯 UI 单例明确限定主线程访问，并在注释和断言中体现；删除或正式使用 `_clipboard_lock`。
  验收标准：跨线程单例有锁；UI 单例只从主线程调用；并发初始化测试覆盖 HooksDLL 和 ClipboardBridge；`_clipboard_lock` 要么被 acquire 要么被移除。

## 15. 不建议近期投入的方向

- 不建议立即做 PySide6 大迁移：当前 PyQt5、Nuitka、hook、窗口特效、兼容层耦合较深，应先完成发布和测试收口。

- 不建议继续无序增加内置命令：先把命令执行抽象、测试矩阵和安全边界固化。
