# 20260606 下一步问题清单

> 范围：本清单只记录问题、风险、证据和架构影响，不写执行顺序、任务拆分或新增功能建议。

## 总体判断

- 当前项目的主要矛盾不是功能少，而是功能面已经很宽，多个高权限能力的边界尚未统一。
- 风险集中在执行模型、插件权限、动作链安全、网络访问、防更新供应链风险、配置事务、线程生命周期、测试真实性。
- 很多问题不是单点 bug，而是同一类架构边界在不同模块重复实现，导致修一个入口并不能覆盖其他入口。

## P0 架构与安全边界问题

### 1. 插件权限模型不是全局强制，默认信任等级可能绕过权限声明

- 证据：
  - `core/plugin_manager.py:136` `PluginManifest.trust_level` 默认是 `local-trusted`。
  - `core/plugin_manager.py:140` manifest 缺少或未知 `trust_level` 时回退为 `local-trusted`。
  - `core/plugin_manager.py:1332` 只有 `m.trust_level == "community-unverified"` 时才启用 `_make_plugin_builtins()` 限制。
  - `core/plugin/constants.py:43` 只在受限 builtins 中阻止 `subprocess`。
  - `tests/test_plugin_manager.py:234` 等测试只覆盖 `community-unverified` 的限制。
- 问题：
  - 外部 `.qlzip` 如果不声明 `trust_level`，会默认成为 `local-trusted`。
  - `local-trusted` 插件不走受限 builtins，理论上可以直接 `import os`、`import subprocess`、读写文件、启动进程。
  - `permissions` 在这种情况下更像声明/提示，而不是强制沙箱。
- 隐藏风险：
  - 用户以为“未声明 process.run/file.write 权限的插件不会执行进程或写文件”，但默认信任路径并不保证这一点。
  - 插件 UI 的高风险确认只看声明的权限；恶意插件可以少声明权限但直接调用 Python 标准库。
- 架构影响：
  - 插件系统的信任模型、权限模型和执行模型没有闭环。
  - 后续做插件生态、插件市场或第三方插件分发前，这会成为最大安全债务。
- 复查结论：
  - 5 条证据全部精确命中，问题定性准确。
- 优化思路：
  - 反转默认信任：`trust_level` 缺失或未知时应回退到最低信任（如 `community-unverified`），而不是 `local-trusted`。升级信任需用户显式确认。
  - 扩展受限 builtins 覆盖：当前 `PLUGIN_BLOCKED_IMPORT_ROOTS` 仅包含 `subprocess`，应将 `ctypes`、`socket`、`multiprocessing` 等也纳入，按权限声明逐项放行。
  - 引入安装来源标记：在 `PluginManifest` 或安装元数据中记录 `install_source`（builtin / official / user_local / third_party），让信任等级由来源+签名+用户选择共同决定。
  - 补全测试矩阵：为 `local-trusted`、`builtin` 等不同 trust level 增加权限限制测试，而不仅覆盖 `community-unverified`。

- 证据：
  - `core/plugin_manager.py:70` 使用全局 `_SHARED_COMMAND_POOL = ThreadPoolExecutor(max_workers=4)`。
  - `core/plugin_manager.py:651` 插件 handler 被提交到共享线程池。
  - `core/plugin_manager.py:652` 使用 `future.result(timeout=PLUGIN_COMMAND_SOFT_TIMEOUT_SECONDS)`。
  - `core/plugin/constants.py:31` 超时时间是 30 秒。
- 问题：
  - `future.result(timeout=...)` 只让等待方返回超时，不能终止正在执行的插件函数。
  - 超时插件仍会继续占住 worker。
  - 4 个 worker 被慢插件或死循环插件占满后，后续插件命令会被拖死。
- 隐藏风险：
  - UI 显示“插件命令超时”，但后台插件代码仍在运行。
  - 插件被隔离时，已经运行中的线程不一定停止。
  - 多个插件共享同一线程池，一个插件的问题可能影响其他插件。
- 架构影响：
  - 插件熔断、隔离、超时、资源释放不是同一个生命周期模型。
  - 这是执行模型问题，不是简单加日志或加超时时间能解决。
- 复查结论：
  - 4 条证据全部精确命中，问题定性准确。补充：实际代码中线程池多了 `thread_name_prefix="plugin-cmd"` 参数，不影响问题本质。
- 优化思路：
  - 进程级隔离：将插件命令执行从线程池迁移到 `multiprocessing.Process` 或 `subprocess`，超时后可真正 `kill`。每个插件分配独立进程，避免共享线程池互相拖死。
  - 协作式取消令牌：为插件 handler 传入 `CancellationToken`，在关键操作（循环、IO、sleep）中检查取消状态，实现软取消。
  - 熔断策略：引入 per-plugin 的熔断器（circuit breaker），超时 N 次后自动熔断，不再提交新任务，避免 4 个 worker 被同一插件反复占满。
  - 隔离执行池：按插件分配独立执行池（或至少 per-plugin 计数），防止一个插件的超时拖垮其他插件的命令执行。

### 3. 插件 HTTP API 没有复用统一网络防护

- 证据：
  - `core/plugin_manager.py:800` `PluginAPI.http_request()` 只校验 `http/https` 和 `netloc`。
  - `core/plugin_manager.py:839` 直接 `urlopen(request, timeout=timeout_value)`。
  - `core/network_security.py:58` 已有 `validate_public_http_url()`。
  - `core/network_security.py:98` 已有 `safe_urlopen()`。
  - `core/network_security.py:132` 已有 `read_limited_response()`。
- 问题：
  - 插件 HTTP 能力和动作链 HTTP 能力使用不同安全边界。
  - 插件请求没有统一拦截 localhost、私网、链路本地地址、保留地址。
  - 插件请求头校验没有复用 `sanitize_headers()`。
- 隐藏风险：
  - 拥有 `network.request` 的插件可能探测本机或内网服务。
  - 修复动作链 HTTP SSRF 不等于修复插件 HTTP SSRF。
- 架构影响：
  - 网络能力没有集中到一个 HostCapability 层。
  - 安全策略散落在多个模块里，后续新增网络入口时容易漏防护。
- 复查结论：
  - 5 条证据全部精确命中，问题定性准确。`PluginAPI.http_request()` 确实未调用 `validate_public_http_url()`、`safe_urlopen()` 和 `read_limited_response()`。
- 优化思路：
  - 直接复用：`PluginAPI.http_request()` 应调用 `safe_urlopen()` 替代裸 `urlopen`，用 `read_limited_response()` 替代自行实现的 `response.read(limit + 1)`。
  - 权限绑定：将 `network.request` 权限与网络 gateway 绑定——只有声明了该权限的插件才能发起 HTTP 请求，且所有请求经过统一拦截链。
  - 响应大小限制统一：当前不同路径的响应大小限制值不同，应定义全局默认值（如 10MB），各场景可覆盖但必须有上限。
  - 审计日志：统一网络 gateway 应记录每次请求的来源模块、URL（脱敏）、响应码、耗时，便于诊断和回溯。

### 4. 动作链危险处理器的 safety 主要停留在定义和 UI 风险提示，运行时强制不足

- 证据：
  - `core/chain/registry.py:151` `_processor_safety()` 定义处理器风险。
  - `core/chain/registry.py:164` `file_write_text`、`folder_create` 被标为 dangerous。
  - `core/chain/enhanced_definitions.py:306` `file_delete` 被标为 dangerous。
  - `ui/config_window/chain_dialog.py:1656` 风险分析在编辑器 UI 中展示。
  - `core/action_chain_host.py:136` `check_permission()` 恒定返回 `True`。
  - `core/action_chain_host.py:143` `request_confirmation()` 恒定返回 `True`。
  - `core/shortcut_chain_exec.py:463` 执行 processor 时直接调用 `execute_chain_processor()`。
- 问题：
  - safety 信息存在，但运行路径没有统一的权限检查和确认拦截。
  - 编辑器中看到风险提示，不代表从快捷方式、命令面板、历史重跑等入口执行时一定会再次确认。
- 隐藏风险：
  - 动作链中写文件、删文件、执行 Python cell、网络请求等能力可能在非编辑器入口静默执行。
  - 后续新增处理器只要登记了 safety，但执行层不强制，就容易出现“标了危险但没拦截”的错觉。
- 架构影响：
  - 动作链 safety 目前更像元数据，不是统一策略执行层。
  - 需要避免把安全逻辑只放在 UI 层，否则会出现多入口绕过。
- 复查结论：
  - 7 条证据中，4.5 `check_permission()` 的 `return True` 实际在第 141 行（文档标注 136 行指向 docstring），4.6 `request_confirmation()` 的 `return True` 在第 156 行（文档标注 143 行指向函数定义），均在函数范围内，定性准确。其余 5 条精确命中。
- 优化思路：
  - 运行时强制拦截层：在 `execute_chain_processor()` 调用前增加统一的 `SafetyInterceptor`，根据 processor 的 safety 定义进行权限检查和用户确认，而非依赖 UI 层。
  - `check_permission()` / `request_confirmation()` 实现化：当前 stub 返回 `True` 应替换为真实实现——检查动作链来源（用户手动/快捷方式/插件调用），对 dangerous 处理器弹出确认对话框，支持"本次会话不再询问"选项。
  - 多入口统一：无论从编辑器、快捷键、命令面板还是历史重跑执行动作链，都必须经过同一个 SafetyInterceptor，确保无绕过路径。
  - 审计链：每次执行 dangerous processor 时记录审计日志（动作链 ID、processor 类型、参数摘要、是否用户确认），便于事后排查。

### 5. 动作链文件处理器具备真实破坏性，但路径边界和删除策略不足

- 证据：
  - `core/chain/processors_files.py:49` `folder_create()` 可创建任意路径。
  - `core/chain/processors_files.py:76` `file_write_text()` 可写入任意路径。
  - `core/chain/processors_files.py:133` `file_delete()` 对文件 `os.remove()`，对目录 `shutil.rmtree()`。
  - `core/chain/enhanced_processors.py:386` 也存在 `shutil.rmtree(path)` 路径。
- 问题：
  - 文件处理器没有运行时路径 allowlist、保护目录检查或回收站策略。
  - `file_delete` 对目录是直接递归删除，不是移动到回收站。
  - 这些能力依赖上层确认，但上层确认本身并不统一。
- 隐藏风险：
  - 用户配置错误、变量绑定错误或插件提供的外部 processor 输入可能导致误删目录。
  - 如果动作链从非 UI 路径执行，危险提示可能缺失。
- 架构影响：
  - 文件系统能力应该属于统一 HostCapability，而不是各 processor 直接调用 `os`/`shutil`。
- 复查结论：
  - 证据 5.2 行号应为 `:76`（原文 `:62`，偏差 14 行），证据 5.3 行号应为 `:133`（原文 `:119`，偏差 14 行），已修正。代码行为描述准确。证据 5.4 中 `enhanced_processors.py:386` 的 `file_delete` 虽有 `to_trash=True` 参数和 `send2trash` 逻辑，但 `send2trash` 导入失败时 fallback 到直接删除，且 `to_trash=False` 时直接走 `shutil.rmtree`，描述准确。
- 优化思路：
  - 路径 denylist：禁止操作 Windows 系统目录（`C:\Windows`、`C:\Program Files` 等）、用户关键目录（`AppData`、`.qoderwork`）、当前 QuickLauncher 安装目录。
  - 删除默认走回收站：`file_delete` 应强制使用 `send2trash`，不提供 `os.remove` / `shutil.rmtree` 的直接回退路径；如果 `send2trash` 不可用，应报错而非降级为永久删除。
  - 路径变量校验：对变量绑定产生的路径进行二次校验，防止变量值为空或包含 `..` 导致路径逃逸。
  - 操作日志：所有文件写/删操作记录到操作日志（路径、操作类型、来源动作链 ID），支持用户查看和撤销。

### 6. 动作链和插件的 timeout 都存在“调用方超时但工作继续”的问题

- 证据：
  - `core/shortcut_chain_exec.py:477` processor timeout 时创建 `ThreadPoolExecutor(max_workers=1)`。
  - `core/shortcut_chain_exec.py:486` `future.result(timeout=timeout_ms / 1000.0)`。
  - `core/plugin_manager.py:652` 插件命令同样使用 future timeout。
- 问题：
  - Python 线程无法被安全强杀。
  - 超时只改变调用方结果，不保证底层写文件、网络请求、进程启动已经停止。
- 隐藏风险：
  - 用户看到“执行超时”，但实际文件写入或外部请求可能仍继续完成。
  - 并发执行时，超时任务继续占用线程，形成资源堆积。
- 架构影响：
  - 超时和取消需要成为执行模型的一等状态，而不是每个模块局部 `future.result(timeout)`。
- 复查结论：
  - 3 条证据全部精确命中，问题定性准确。
- 优化思路：
  - 引入 `CancellationToken` 一等公民：所有可超时的执行路径（插件命令、动作链 processor、搜索）统一接受 token 参数，超时后设置取消标志，让底层操作有机会提前退出。
  - 统一任务包装器：创建 `CancellableTask` 抽象，封装提交、超时、取消、结果获取，替代各模块分散的 `ThreadPoolExecutor + future.result(timeout)` 模式。
  - 超时后资源回收：超时任务虽然线程不能强杀，但可以关闭其持有的资源（socket、文件句柄），减少副作用继续执行的可能性。
  - 超时事件通知：超时不仅改变调用方返回值，还应触发事件通知 UI 更新、审计日志和可能的熔断器计数。

### 7. IPC 本地实例通道没有鉴权

- 证据：
  - `main.py:217` 使用固定 server name `QuickLauncherInstance_v3`。
  - `bootstrap/ipc.py:42` 创建 `QLocalServer`。
  - `bootstrap/ipc.py:66` 客户端发送固定文本 `show_config`。
  - `bootstrap/ipc.py:31` 服务端直接解析命令。
- 问题：
  - 任意同用户进程理论上都可以连接本地 IPC 并发送 `show_config`。
  - 当前命令只有打开配置窗口，破坏性有限，但通道没有为未来扩展留下鉴权基础。
- 隐藏风险：
  - 如果未来 IPC 增加“执行命令”“重载插件”“打开文件”等动作，固定无鉴权通道会变成高风险入口。
  - 恶意本地进程可反复触发窗口，造成打扰或 DoS。
- 架构影响：
  - IPC 命令系统应默认具备 session token/nonce/命令白名单，而不是后补。
- 复查结论：
  - 4 条证据全部精确命中，问题定性准确。
- 优化思路：
  - 启动时生成随机 session token 写入临时文件：客户端连接后先发送 token，服务端验证后才接受后续命令。token 随进程退出自动失效。
  - 命令白名单：当前只有 `show_config`，但应预设命令注册表，每个命令声明权限级别（safe / privileged），privileged 命令需要额外校验。
  - 频率限制：对连续连接请求做 rate limiting（如每秒最多 5 次），防止恶意本地进程反复触发。
  - 协议版本化：当前是裸文本协议，建议改为 JSON 格式 `{token, command, params}`，便于扩展和校验。

### 8. 更新链路只有 hash 校验，没有签名信任边界

- 证据：
  - `services/update/checker.py:224` 从 GitHub asset digest 或 release body 提取 `sha256`。
  - `services/update/checker.py:249` 要求有效 sha256。
  - `services/update/downloader.py:134` 下载后校验 sha256。
  - `services/update/installer.py:55` 安装前再次校验 sha256。
- 问题：
  - sha256 能防传输损坏或下载目标被替换，但如果 release 元数据本身被篡改，hash 也会一起被篡改。
  - 没看到独立签名、公钥 pinning、Authenticode 校验或 release manifest 签名。
- 隐藏风险：
  - GitHub release 权限或 token 被攻破时，攻击者可同时发布恶意 exe 和对应 hash。
  - 用户会看到 hash 校验通过，但这不等同于发布者签名验证。
- 架构影响：
  - 自动更新是供应链边界，需要独立于下载渠道的信任根。
- 复查结论：
  - 4 条证据全部精确命中，问题定性准确。当前更新链路在 checker、downloader、installer 三处均有 sha256 校验，但缺少独立于下载渠道的签名验证。
- 优化思路：
  - Ed25519 签名校验：在 CI 构建时，用私钥对安装包 + hash 签名；应用内置公钥，安装前验证签名。私钥只存于 CI secret，不进入源码。
  - 公钥 pinning：将公钥硬编码在应用内（如 `core/update/trust.py`），不依赖外部配置或系统证书链。
  - Release manifest 签名：发布时在 GitHub release 中附带 `manifest.json.sig`，包含每个 asset 的 hash 和整体签名，checker 先验签名再验 hash。
  - Authenticode 签名：对 Windows 安装包 (.exe) 增加代码签名证书（如 EV Code Signing），让用户系统和 SmartScreen 也能验证发布者身份。

### 9. 网络防护实现分散，多套逻辑可能出现策略漂移

- 证据：
  - `core/network_security.py` 有共享防护。
  - `core/favicon_cache.py:417` 维护了一套独立 `_safe_urlopen()` 和 URL 校验。
  - `services/update/downloader.py:96` 直接使用 `urlopen()`。
  - `services/api/base_client.py:61` 直接使用 `urlopen()`。
  - `core/plugin_manager.py:839` 插件 HTTP 直接使用 `urlopen()`。
  - `core/command_variables.py:200` WAN IP 获取直接使用 `urllib.request.urlopen()`。
- 问题：
  - 不同模块各自处理 URL、重定向、host、大小限制、header。
  - 有些路径有私网拦截，有些没有。
  - 有些路径限制响应大小，有些没有或限制不同。
- 隐藏风险：
  - 修复一个 SSRF 入口后，另一个入口仍存在。
  - 测试覆盖某个网络模块不代表网络能力整体安全。
- 架构影响：
  - 网络访问应该通过统一 gateway，以 capability 区分 favicon、update、plugin、action-chain、diagnostics 等场景。
- 复查结论：
  - 6 条证据中，`base_client.py` 的 `urlopen` 行号已从 `:55` 修正为 `:61`。补充发现：`network_security.py` 目前只被 `core/chain/processors_network.py` 一个模块导入使用，其他 5 处网络入口均未复用。问题定性准确且比文档描述更严重。
- 优化思路：
  - 创建 `NetworkGateway` 统一入口：所有 HTTP 请求通过 `gateway.request(url, *, capability, timeout, max_bytes)` 发起，内部统一调用 `validate_public_http_url()` + `safe_urlopen()` + `read_limited_response()`。
