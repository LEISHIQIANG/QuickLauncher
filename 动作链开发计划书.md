# QuickLauncher 动作链开发计划书

## 1. 目标与边界

动作链的目标不是把 QuickLauncher 变成复杂自动化平台，而是在现有“快捷方式执行器”之上增加轻量编排能力：一个图标可以按顺序执行多个已有快捷方式，覆盖“打开项目环境”“启动一组工具”“先打开程序再发送快捷键”“打开目录后执行命令”等高频场景。

第一版必须优先复用当前架构，避免重写执行体系：

- 数据模型继续落在 `core/data_models.py` 的 `ShortcutItem`。
- 执行入口继续走 `core/shortcut_executor.py` 的 `ShortcutExecutor.execute()`。
- 文件、URL、命令、快捷键执行逻辑继续复用现有 mixin：`shortcut_file_exec.py`、`shortcut_url_exec.py`、`shortcut_command_exec.py`、`shortcut_hotkey.py`。
- 弹窗点击执行仍由 `ui/launcher_popup/popup_item_execution.py` 触发。
- 配置保存、历史、导入恢复继续走 `core/data_manager.py` 和 `core/config_validation.py`。

动作链第一版建议只做“引用已有快捷方式 + 延迟 + 失败策略 + 执行摘要”。不要在第一版加入条件分支、变量传递、并行执行和复杂脚本化能力。这些能力容易把功能范围推大，也会放大命令安全、UI 编辑器和测试成本。

## 2. 当前项目衔接点

### 2.1 数据模型

当前快捷方式类型定义在 `core/data_models.py`：

```python
class ShortcutType(Enum):
    FILE = "file"
    FOLDER = "folder"
    URL = "url"
    HOTKEY = "hotkey"
    COMMAND = "command"
```

`ShortcutItem.to_dict()` 和 `ShortcutItem.from_dict()` 负责所有快捷方式字段序列化。新增动作链类型后，配置文件会自然保存到 `config/data.json`。`core/config_validation.py` 使用 `{item.value for item in ShortcutType}` 生成合法类型集合，因此只要把 `CHAIN = "chain"` 加进枚举，配置校验会自动认可新类型。

需要注意：`DataManager.export_shareable_config()` 和 `_import_shareable_config_safe()` 当前只处理 `hotkey`、`command`、`url`。动作链是否可分享需要单独设计，因为链内可能引用本机已有快捷方式 ID。第一版建议不纳入“可分享配置”，或者导出时只导出链本体并把引用断开为无效步骤，容易造成体验不佳。更稳的方案见第 9 节。

### 2.2 执行入口

当前执行调度在 `core/shortcut_executor.py`：

```python
ShortcutExecutor.execute(shortcut, force_new=False)
```

它按类型分发到 hotkey、url、command、file/folder。动作链应该作为新分支加入：

```python
elif shortcut.type == ShortcutType.CHAIN:
    return ShortcutExecutor._execute_chain(shortcut, force_new)
```

建议新增模块 `core/shortcut_chain_exec.py`，实现 `ChainExecutionMixin`，再把它加入 `ShortcutExecutor` 继承列表与 `_bind_shortcut_executor_mixins()`。

这样保持现有 mixin 风格一致，也方便单独测试。

### 2.3 弹窗执行与运行时输入

弹窗点击逻辑在 `ui/launcher_popup/popup_item_execution.py`。当前逻辑只在 `COMMAND` 和 `URL` 上收集 `{input}`，文件拖放只对 `FILE/FOLDER` 生效。

动作链第一版不需要在弹窗层特殊处理运行时输入，原因是：

- 链内如果引用命令快捷方式，真正执行时现有 `ShortcutExecutor.execute()` 会进入命令执行逻辑。
- 但当前 `{input}` 的收集发生在弹窗层，而不是执行器层。链内命令若含 `{input}`，直接从链执行器调用 `ShortcutExecutor.execute()` 时不会弹输入框。

因此第一版需要明确一个限制：动作链 MVP 不支持链内步骤运行时输入。更好的工程方案是把运行时输入收集抽成通用服务，例如 `core/runtime_inputs.py` 或 UI 回调接口，但这会牵涉 UI 线程和执行线程，不建议第一阶段做。

### 2.4 使用统计

`core/data_manager.py` 中 `record_shortcut_used(shortcut_id)` 记录使用次数和最近使用时间，并支持智能排序。动作链执行时建议：

- 用户点击的动作链本体成功启动后记录一次使用。
- 链内被引用的快捷方式默认也记录使用，因为它确实被执行了。
- 如果用户认为链内统计污染智能排序，可在后续增加 `chain_record_step_usage: bool` 设置。第一版先记录，符合直觉。

## 3. 数据结构设计

### 3.1 ShortcutType

新增：

```python
class ShortcutType(Enum):
    FILE = "file"
    FOLDER = "folder"
    URL = "url"
    HOTKEY = "hotkey"
    COMMAND = "command"
    CHAIN = "chain"
```

### 3.2 ShortcutItem 新增字段

建议字段：

```python
chain_steps: list[dict] = field(default_factory=list)
chain_stop_on_error: bool = True
chain_show_summary: bool = True
chain_step_timeout_seconds: float = 30.0
```

第一版不要加太多字段。步骤级超时目前对大部分现有执行器不一定能强制终止，因为文件、URL 和显示窗口命令都是“启动即返回”。但它可以用于未来捕获输出命令或插件命令。第一版可以保留字段但不强制实现，或直接推迟。

`to_dict()` 增加：

```python
"chain_steps": self.chain_steps,
"chain_stop_on_error": self.chain_stop_on_error,
"chain_show_summary": self.chain_show_summary,
"chain_step_timeout_seconds": self.chain_step_timeout_seconds,
```

`from_dict()` 增加：

```python
item.chain_steps = _sanitize_chain_steps(data.get("chain_steps", []))
item.chain_stop_on_error = bool(data.get("chain_stop_on_error", True))
item.chain_show_summary = bool(data.get("chain_show_summary", True))
item.chain_step_timeout_seconds = float(data.get("chain_step_timeout_seconds", 30.0) or 30.0)
```

建议把 `_sanitize_chain_steps()` 放在 `ShortcutItem` 静态方法或新模块 `core/chain_models.py` 中，避免污染 `from_dict()`。

### 3.3 ChainStep 格式

MVP 只支持引用已有快捷方式：

```json
{
  "id": "9b3f2b0a-0ad3-48a9-901b-7a07d25eb222",
  "kind": "shortcut_ref",
  "shortcut_id": "target-shortcut-id",
  "name": "打开项目目录",
  "enabled": true,
  "delay_ms": 300,
  "on_error": "default"
}
```

字段说明：

- `id`：步骤自身 ID，用于排序、日志、未来变量引用。
- `kind`：第一版固定为 `shortcut_ref`。
- `shortcut_id`：引用的已有快捷方式 ID。
- `name`：步骤显示名。编辑器中可默认同步被引用快捷方式名称，但允许用户改别名。
- `enabled`：是否启用该步骤。
- `delay_ms`：执行该步骤前等待多久。范围建议 `0-60000`。
- `on_error`：第一版可保留但只支持 `default`。未来支持 `stop`、`continue`、`ask`。

不要在第一版把 file/url/command/hotkey 的全部字段复制进 `chain_steps`。这样会带来重复编辑器、导入导出、图标、变量和权限逻辑。先引用已有快捷方式，能最大化复用现有稳定能力。

## 4. 执行器设计

### 4.1 新增模块

新增 `core/shortcut_chain_exec.py`。

职责：

- 解析 `shortcut.chain_steps`。
- 查找被引用的 `ShortcutItem`。
- 检测循环引用。
- 按顺序执行步骤。
- 汇总执行结果。
- 返回 `(success, error_message)`，保持 `ShortcutExecutor.execute()` 当前契约。

建议核心函数：

```python
class ChainExecutionMixin:
    @staticmethod
    def _execute_chain(shortcut: ShortcutItem, force_new: bool = False) -> tuple[bool, str]:
        ...
```

辅助函数：

```python
def _find_shortcut_by_id(shortcut_id: str) -> ShortcutItem | None
def _iter_enabled_chain_steps(shortcut: ShortcutItem) -> list[dict]
def _detect_chain_cycle(root_id: str, next_id: str, visited: set[str]) -> bool
def _build_chain_summary(results: list[ChainStepResult]) -> str
```

可以新增 dataclass：

```python
@dataclass
class ChainStepResult:
    step_id: str
    step_name: str
    shortcut_id: str
    success: bool
    skipped: bool = False
    error: str = ""
    duration: float = 0.0
```

### 4.2 查找快捷方式

当前 `DataManager` 没有公开 `get_shortcut_by_id()`。建议增加：

```python
def get_shortcut_by_id(self, shortcut_id: str) -> ShortcutItem | None:
    for folder in self.data.folders:
        for item in folder.items:
            if item.id == shortcut_id:
                return item
    return None
```

也可以先在 `shortcut_chain_exec.py` 内部遍历 `core.data_manager.data.folders`，但长期看放在 `DataManager` 更合适，因为后续健康检查、动作链编辑器、导入导出都会用到。

### 4.3 循环引用

动作链可以引用动作链，因此必须防止循环。

示例：

```text
A 引用 B
B 引用 A
```

执行时维护栈：

```python
_execute_chain(shortcut, chain_stack=None)
```

如果当前 `shortcut.id` 已在 `chain_stack` 中，返回失败：

```text
动作链存在循环引用: A -> B -> A
```

为了不改变 `ShortcutExecutor.execute()` 的签名，可以把内部调用改为私有 helper：

```python
ShortcutExecutor._execute_shortcut_internal(item, force_new, chain_stack)
```

但这会动主执行器。第一版更简单的办法是：`_execute_chain()` 只在执行前递归扫描链引用，确认无环后，再逐步调用 `ShortcutExecutor.execute()`。扫描函数直接访问数据模型，不执行动作。

### 4.4 失败策略

第一版全局字段 `chain_stop_on_error`：

- `True`：某一步失败后停止后续步骤。
- `False`：继续执行后续步骤，但最终返回失败。

最终成功规则：

- 所有启用步骤成功：动作链成功。
- 存在失败步骤：动作链失败。
- 没有启用步骤：动作链失败，错误为“动作链没有可执行步骤”。
- 引用失效：视为失败步骤。

### 4.5 延迟

每步执行前应用 `delay_ms`：

```python
if delay_ms > 0:
    time.sleep(delay_ms / 1000)
```

注意这发生在后台线程中，因为弹窗点击后 `_execute_item()` 已经用 `threading.Thread(target=do_execute_thread)` 执行，不会阻塞 UI。

### 4.6 面板结果

当前普通快捷方式执行失败会弹错误框，命令面板结果只对 panel 命令生效。动作链如果只返回 `(False, summary)`，目前可能会走 `_on_execution_error()` 的错误弹窗，不适合展示多步骤摘要。

有两个方案：

1. 第一版简单处理：失败时弹错误框，成功时不展示摘要。
2. 更契合命令增强路径：让动作链执行器把 `CommandResult` 写入 `command_registry.set_pending_command_result()`，弹窗层检测并展示。

建议选择第二种，但需要配合“命令增强优化”文档中的统一结果通道。动作链结果可以是：

```python
CommandResult(
    success=success,
    display_type="list",
    message=summary,
    payload={"items": [...]},
    actions=[CommandAction(type="copy", label="复制摘要", value=summary)]
)
```

如果命令增强暂未实现，动作链 MVP 可先记录日志并返回错误字符串。文档实施顺序建议先做命令结果通道增强，再做动作链摘要。

## 5. UI 设计

### 5.1 新增动作链编辑器

建议新增文件：

```text
ui/config_window/chain_dialog.py
```

不要塞进 `command_dialog.py`。动作链不是命令，独立对话框更清晰。

编辑器组成：

- 基本信息：名称、图标、标签。
- 链设置：失败时停止、执行后显示摘要。
- 步骤列表：显示步骤名、引用类型、延迟、启用状态。
- 操作按钮：添加已有快捷方式、上移、下移、删除、测试运行。

第一版步骤列表可以用 `QListWidget` 或 `QTableWidget`。考虑现有配置窗口已有批量操作和拖拽排序，长期可以做拖拽列表，但 MVP 用上移/下移按钮更稳。

### 5.2 添加已有快捷方式

需要一个选择器，列出所有非当前动作链的快捷方式：

显示内容：

```text
分类 / 名称 / 类型 / 目标摘要
```

过滤规则：

- 不显示当前动作链本身。
- 可以显示其他动作链，但保存前和执行前都要做循环检测。
- 默认不显示 disabled 项，或者显示但标记“已禁用”。建议显示并标记，避免用户疑惑找不到。

可以先实现为简单 `QInputDialog` 或 `QDialog + QLineEdit + QListWidget`。为了可用性，建议支持搜索名称、别名、标签。

### 5.3 设置窗口入口

当前添加快捷方式入口分散在 `ui/config_window/main_window.py`、`settings_panel.py` 等模块。动作链需要：

- 底部新增“动作链”按钮，或添加菜单中新增“动作链”。
- 双击动作链图标时打开 `ChainDialog`。
- 图标渲染可以先用默认命令/链图标，后续加内置图标。

需要检查并更新所有基于类型判断的 UI：

- 添加按钮逻辑。
- 编辑逻辑。
- 复制/移动/批量启用禁用逻辑。
- 图标预览和默认图标。
- 搜索结果展示中类型名称。

### 5.4 弹窗展示

动作链作为普通快捷方式显示在图标网格中。建议图标默认使用“链条/闪电/流程”类内置图标。项目当前已有 `assets/command_icons`，可新增 `chain.png`，或者先用现有 `restart.png`/`config.png` 临时代替，不建议为了第一版阻塞。

## 6. 配置、导入导出与兼容性

### 6.1 普通配置保存

`config/data.json` 会随 `ShortcutItem.to_dict()` 自动保存。新增字段必须有默认值，旧配置加载不会受影响。

### 6.2 配置历史

`DataManager` 的历史快照记录基于完整 data dict。动作链字段会自动纳入历史，不需要额外工作。

### 6.3 配置校验

`config_validation.py` 中合法类型自动来自 `ShortcutType`。但建议增加动作链步骤基础校验：

- `chain_steps` 必须是 list。
- 单个动作链最多 128 步。
- `delay_ms` 限制在 `0-60000`。
- `kind` 非法时跳过。
- `shortcut_id` 为空时跳过或标记无效。

可以在 `ShortcutItem.from_dict()` 里做轻量清洗，也可以扩展 `sanitize_app_data_dict()`。优先放在 `ShortcutItem.from_dict()`，改动小。

### 6.4 可分享配置

当前分享配置只导出 `hotkey`、`command`、`url`，不导出 `file`，因为文件路径强本机绑定。动作链的引用也强绑定本机 ID。

第一版建议：不导出动作链。

第二版方案：

- 导出动作链时同时导出被引用的 `hotkey/command/url` 项。
- 建立旧 ID 到新 ID 的映射。
- 导入后重写 `chain_steps[].shortcut_id`。
- 如果链引用 file/folder 项，则跳过该步骤并在导入报告中提示。

## 7. 安全与风险控制

动作链会把多个动作组合到一起，风险不是新权限，而是“一次点击触发更多影响”。必须做风险摘要：

保存或执行前可计算：

- 是否包含管理员运行步骤。
- 是否包含命令步骤。
- 是否包含 Python legacy inline。
- 是否包含外部 URL。
- 是否包含快捷键发送。

第一版建议在编辑器中展示静态提示，不做强拦截：

```text
此动作链包含 1 个命令步骤、1 个快捷键步骤。请确认这些步骤来自可信配置。
```

如果包含 `run_as_admin=True`，执行时沿用现有 UAC 机制，不自行绕过。

## 8. 测试计划

### 8.1 单元测试

新增：

```text
tests/test_shortcut_chain_exec.py
```

覆盖：

- 空动作链返回失败。
- 单步骤成功。
- 多步骤按顺序执行。
- 步骤 disabled 时跳过。
- 引用不存在时失败。
- `chain_stop_on_error=True` 时停止后续步骤。
- `chain_stop_on_error=False` 时继续后续步骤但最终失败。
- 延迟值被清洗到安全范围。
- 循环引用检测。
- 动作链字段 `to_dict/from_dict` 往返。

执行器测试可以 monkeypatch `ShortcutExecutor.execute` 或更底层的 `_execute_file/_execute_url`，避免真实启动程序。

### 8.2 配置测试

扩展：

- `tests/test_config_validation.py`
- `tests/test_data_manager.py`

覆盖：

- `chain` 类型被配置校验认可。
- 旧配置缺少动作链字段时正常加载。
- 非法 `chain_steps` 被清洗。
- 重复快捷方式 ID 检测仍正常。

### 8.3 UI 测试

如果现有 UI 测试环境可用，增加轻量测试：

- 新建 `ChainDialog` 不崩溃。
- 加载已有动作链显示步骤。
- 保存后字段正确。

不要在第一版做复杂拖拽 UI 测试。

## 9. 分阶段实施

### 阶段 1：核心模型与执行器

改动文件：

- `core/data_models.py`
- `core/shortcut_executor.py`
- `core/shortcut_chain_exec.py`
- `core/data_manager.py`
- `core/config_validation.py`
- `tests/test_shortcut_chain_exec.py`

交付：

- `ShortcutType.CHAIN`
- 动作链字段序列化。
- 引用已有快捷方式执行。
- 循环检测。
- 失败停止/继续。
- 单元测试。

### 阶段 2：基础 UI

改动文件：

- `ui/config_window/chain_dialog.py`
- `ui/config_window/main_window.py`
- `ui/config_window/settings_panel.py`
- 图标相关模块按实际入口补齐。

交付：

- 可创建、编辑、删除动作链。
- 可添加已有快捷方式为步骤。
- 可调整步骤顺序。
- 可设置延迟和失败策略。
- 可测试运行。

### 阶段 3：结果面板衔接

依赖命令增强路径。

交付：

- 动作链执行后显示列表摘要。
- 失败步骤可复制错误详情。
- 结果面板支持“重新执行动作链”。

### 阶段 4：分享配置与高级能力

交付：

- 动作链分享导出。
- 导入时重写引用 ID。
- 支持内联步骤。
- 支持步骤级失败策略。
- 支持链变量和上一步输出。

## 10. 不建议第一版做的功能

- 条件分支。
- 并行执行。
- 循环/重复 N 次。
- 链内直接编辑完整 command/url/file/hotkey 字段。
- 链内 `{input}` 运行时输入。
- 强制终止任意外部程序。
- 独立动作链脚本语言。

这些都可以做，但它们会把动作链从“快捷方式编排”推成“自动化引擎”。QuickLauncher 当前最自然的演进路径是轻量、可解释、可测试的动作编排。

## 11. 验收标准

动作链第一版完成后，应满足：

- 用户能创建一个动作链并加入 2-5 个已有快捷方式。
- 点击动作链后步骤按顺序执行。
- 某一步失败时能按设置停止或继续。
- 链引用不存在时不会崩溃，并给出可理解错误。
- 循环引用不会导致无限递归。
- 旧配置可正常加载。
- 配置保存、历史、恢复不破坏动作链字段。
- 核心逻辑有单元测试覆盖。

## 12. 2026-05-26 修订：动作链结果进入独立命令面板

本计划原先提到“动作链执行摘要可显示在结果面板”。现在产品决策调整为：

```text
中键弹窗只作为入口。
点击动作链后，中键弹窗立即消失。
动作链的执行状态、每步结果、错误详情、复制报告、重新执行入口，统一进入独立命令面板。
```

因此，动作链路径需要与 `命令面板独立化优化计划书.md` 对齐：

- 动作链仍然是新的 `ShortcutType.CHAIN`。
- 动作链点击后由 `PopupItemExecutionMixin._execute_item()` 触发，但弹窗应立即隐藏。
- 动作链执行器生成 `CommandResult(display_type="list")`。
- 完整结果写入 `CommandResultStore`。
- `CommandPanelWindow` 显示动作链执行状态和步骤列表。
- 不在 `popup_command_result.py` 内继续扩展动作链报告 UI。

动作链结果建议结构：

```python
CommandResult(
    success=success,
    display_type="list",
    message="动作链完成 4/5 步，1 步失败",
    payload={
        "items": [
            {
                "title": "打开项目目录",
                "status": "success",
                "detail": "C:/Work/App",
                "duration": 0.12,
            },
            {
                "title": "执行 git status",
                "status": "failed",
                "detail": "工作目录不存在",
                "duration": 0.03,
            },
        ]
    },
    actions=[
        CommandAction(type="copy", label="复制报告", value=report_text),
        CommandAction(type="retry", label="重新执行", value=chain_id),
    ],
)
```

动作链实施顺序调整为：

```text
命令面板独立化
CommandResultStore
动作链核心模型与执行器
动作链编辑器
动作链结果写入命令面板
动作链分享导入导出
动作链变量和条件
```

动作链第一版验收标准同步增加：

- 点击动作链后，中键弹窗立即消失。
- 动作链执行时不依赖中键弹窗存在。
- 成功、失败、跳过步骤都能在独立命令面板中查看。
- 失败步骤可复制详情。
- 可从命令面板重新执行动作链。

## 13. 2026-05-26 修订：动作链入口与编辑窗口布局

动作链功能第一阶段不放到配置窗口的显眼主按钮区，先隐藏在配置面板图标栏空白位置的右键菜单中。这样可以让动作链作为高级功能灰度进入，不打扰现有添加快捷方式、URL、命令、快捷键的主流程。

### 13.1 临时入口

入口规则：

```text
配置窗口
  -> 图标栏空白位置右键
  -> 菜单项：新建动作链
  -> 打开 ChainDialog
```

实现位置需要结合现有配置窗口代码确认，优先检查：

```text
ui/config_window/main_window.py
ui/config_window/icon_grid.py
ui/config_window/settings_panel.py
```

要求：

- 只在图标栏空白位置右键菜单中出现，不在第一版添加到底部主按钮。
- 右键在已有图标上时仍保持现有图标菜单行为，不混入“新建动作链”。
- 通过该入口创建的动作链仍作为普通 `ShortcutItem(type=CHAIN)` 写入当前分类。
- 后续功能稳定后，再考虑加入显式按钮或添加菜单。

### 13.2 动作链编辑窗口布局

动作链窗口采用左右两栏布局，不使用单列表单式窗口。

建议文件：

```text
ui/config_window/chain_dialog.py
```

窗口布局：

```text
┌─────────────────────────────────────────────────────────────┐
│ 标题栏：动作链                                               │
├───────────────────────────────┬─────────────────────────────┤
│ 左侧：编排区                   │ 右侧：执行结果区             │
│                               │                             │
│ [顶部] 基础模块                │ 当前选中卡片 / 最近执行结果  │

│ - 失败策略                     │ - 错误信息                  │
│ - 执行后打开命令面板           │ - 耗时                      │
│                               │                             │
│ [左中] 命令执行卡片列表         │                             │

```

左侧顶部为动作链基础模块，左中和左下为命令执行卡片区域，右侧一栏专门显示每个卡片的执行结果。

### 13.3 命令执行卡片

每个步骤在左侧显示为一张执行卡片，而不是普通表格行。卡片需要展示：

- 步骤名称。
- 步骤类型：引用快捷方式、等待、后续内联命令等。
- 启用/禁用状态。
- 延迟时间。
- 失败策略。
- 被引用快捷方式的目标摘要。

卡片操作：

- 单击卡片：右侧显示该卡片最近一次执行结果。
- 双击卡片：编辑步骤。
- 右键卡片：禁用、复制、删除、上移、下移。

第一版卡片可以用 `QFrame`/`QWidget` 自绘或 `QListWidget` item widget 实现，优先选择维护成本低、与现有主题样式容易统一的方案。

### 13.4 右侧执行结果区

右侧结果栏不是最终完整命令面板的替代品，它只服务于动作链编辑与测试。

显示内容：

- 当前选中卡片的最近测试结果。
- 成功、失败、跳过状态。
- 错误详情。
- 耗时。
- 命令输出摘要。
- “打开完整命令面板”入口。

正式运行动作链时，完整结果仍进入独立 `CommandPanelWindow`，不进入中键弹窗。

### 13.5 UI 风格约束

动作链窗口需要参考现有配置窗口和工具窗口样式：

- 基础窗口风格优先复用 `BaseDialog` 或现有配置窗口风格。
- 卡片圆角控制在项目现有风格范围内，不做过度装饰。
- 按钮、输入框、列表、滚动条尽量复用 `ui.styles.style`、`theme_helper` 和已有 config window 工具函数。
- 不引入新的大面积配色体系。
- 文本密度应接近配置窗口，不做营销式大标题。

## 14. 与三条优化路径的顺序关系

动作链应排在命令面板独立化和命令增强之后实现。推荐顺序：

```text
1. 命令面板独立化
2. 命令增强
3. 动作链
```

动作链依赖前两项的原因：

- 动作链执行后需要展示每一步的状态、错误和输出摘要，这依赖独立 `CommandPanelWindow`，不应回退到中键弹窗。
- 动作链步骤可能包含 CMD/Python 命令，若这些命令开启捕获输出，需要复用命令增强的捕获输出和 `CommandResult` 协议。
- 动作链结果适合 `display_type=list`，该类型应先在命令面板中稳定。
- 动作链窗口右侧“每个卡片的执行结果”可以复用命令增强的结果摘要格式，避免自己定义另一套展示协议。

动作链开工前应确认这些前置能力已经完成：

- `CommandResultStore` 可用。
- `CommandPanelWindow` 可显示 `list/log/text`。
- 命令面板有 `large` 和 `auto` 窗口模板。
- `TrayApp.show_command_panel()` 可由弹窗和执行器调用。
- 普通命令捕获输出返回标准 `CommandResult`。

动作链第一版只做：

```text
图标栏空白处右键入口
左右两栏 ChainDialog
引用已有快捷方式步骤
顺序执行
失败停止/继续
结果写入命令面板
```

不要在第一版做：

- 动作链自定义结果窗口。
- 复杂条件分支。
- 并行执行。
- 上一步输出变量流转。
- 分享导入导出。

这些能力等命令面板、命令增强和基础动作链都稳定后再推进。
