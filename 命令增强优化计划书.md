# QuickLauncher 命令增强优化计划书

## 1. 目标与边界

命令增强的目标是把 QuickLauncher 现有“命令执行 + 插件结果面板”打磨成稳定的本地命令中心。它应该解决三个问题：

1. 普通 CMD/Python 命令执行后缺少可读反馈。
2. 插件 `CommandResult` 已有结构化字段，但 UI 渲染能力和交互动作还不完整。
3. 长任务、进度、取消、参数输入目前没有统一协议。

这条路径必须尊重当前项目已经形成的边界：

- `core/command_registry.py` 已经是命令定义、上下文、结果模型的中心。
- `core/shortcut_command_exec.py` 是普通命令快捷方式执行入口。
- `ui/launcher_popup/popup_command_result.py` 是结果面板渲染入口。
- `core/plugin_manager.py` 已经校验插件返回的 `CommandResult` 字段。
- `core/commands.py` 和 `plugins/*/main.py` 已经大量返回 `CommandResult`。

因此本计划不是另起一套命令系统，而是补齐当前 Command Center 2.0 的协议、执行通道和 UI 渲染。

## 2. 当前项目检查结论

### 2.1 已有能力

`core/command_registry.py` 已有模型：

```python
@dataclass
class CommandAction:
    type: str = "copy"
    label: str = ""
    value: str = ""

@dataclass
class CommandContext:
    raw_input: str = ""
    args_text: str = ""
    args: dict[str, str] = field(default_factory=dict)
    clipboard_text: str = ""
    selected_files: list[str] = field(default_factory=list)
    update_callback: Callable[[CommandResult], None] | None = None

@dataclass
class CommandResult:
    success: bool = True
    message: str = ""
    display_type: str = "text"
    payload: dict[str, Any] = field(default_factory=dict)
    actions: list[CommandAction] = field(default_factory=list)
    error: str = ""
    is_async: bool = False
    progress: float = 0.0
    cancellable: bool = False
```

说明项目已经预留了结构化结果、异步进度和取消字段。

弹窗面板 `ui/launcher_popup/popup_command_result.py` 已支持：

- 文本结果。
- 简单表格。
- QR 图片。
- 进度条的一部分逻辑。
- 最多两个 action 按钮。
- copy/open_url/open_file/open_folder/save_text/save_file/create_shortcut 等 action 类型。

插件系统 `core/plugin_manager.py` 已支持插件返回 dict 或 `CommandResult`，并保留以下字段：

```text
success, message, display_type, payload, actions, error, is_async, progress, cancellable
```

### 2.2 主要缺口

当前缺口集中在五处：

1. **普通 CMD/Python 快捷方式没有统一结果回显**
   `ShortcutExecutor._execute_command()` 多数时候只返回 `(success, error)`，静默命令不捕获 stdout/stderr 给面板。

2. **结果通道是单槽 pending result**
   `command_registry.set_pending_command_result()` / `take_pending_command_result()` 是全局单槽，只适合短期桥接，不适合多任务、并发、异步进度。

3. **`display_type` 没有完整协议**
   UI 只显式处理 `table`、`qr` 和普通文本，`kv/list/log/progress/confirm` 等类型没有规范。

4. **Action 数量固定截断**
   `MAX_COMMAND_RESULT_ACTIONS = 2` 可以保护紧凑面板，但缺少“更多”菜单，复杂命令只能牺牲操作。

5. **参数定义未充分使用**
   `CommandDefinition.params` 已存在，但当前命令执行主要依赖 `args_text` 和插件自行解析字符串。

另外，`ui/launcher_popup/popup_item_execution.py` 中读取了 `settings.enable_command_panel_v2`，但 `AppSettings` 当前未定义该字段。由于使用 `getattr` 包裹在 try 中，运行上可能不会崩，但这属于历史遗留开关，命令增强时应清理：要么正式加入设置，要么移除此判断并默认启用当前面板逻辑。

## 3. 设计原则

1. **先增强协议，再扩大功能**
   先把 `CommandResult` 的 display/payload/action 约定写清并实现 UI，再改造插件和命令。

2. **普通命令和插件命令走同一种结果面板**
   用户不应该感知“这是快捷方式命令”还是“插件命令”，只看结果。

3. **长任务必须可观察**
   文件哈希、磁盘扫描、事件日志、网络请求都应能显示进度或至少显示“正在执行”。

4. **紧凑面板优先**
   QuickLauncher 弹窗空间有限。默认结果面板保持紧凑，复杂结果通过滚动、展开或更多菜单解决。

5. **保留安全边界**
   命令捕获输出不应鼓励任意 shell 注入。现有 `{clipboard:q}`、变量安全检查和 `command_risk.py` 必须继续生效。

## 4. CommandResult 协议升级

### 4.1 display_type 规范

建议正式支持以下类型：

```text
text      普通文本，默认类型
log       长文本日志，等同 text 但使用等宽字体和更大的滚动区域
table     表格
kv        键值信息
list      列表
progress 进度状态
confirm   二次确认
qr        已有二维码类型，保持兼容
```

### 4.2 payload 结构

#### text/log

```python
payload = {
    "monospace": True,
    "wrap": False,
    "max_lines": 200
}
```

#### table

当前 UI 只读取 `payload["rows"]`。建议扩展为：

```python
payload = {
    "columns": ["PID", "Name", "CPU", "Memory"],
    "rows": [
        [1234, "python.exe", "12.5%", "240 MB"]
    ],
    "copy_format": "tsv"
}
```

兼容策略：如果没有 `columns`，继续按旧逻辑只画 rows。

#### kv

```python
payload = {
    "items": [
        ["IPv4", "192.168.1.8"],
        ["Gateway", "192.168.1.1"],
        ["DNS", "223.5.5.5"]
    ]
}
```

#### list

```python
payload = {
    "items": [
        {"title": "打开项目目录", "status": "success", "detail": "C:/Work/App"},
        {"title": "执行 git status", "status": "failed", "detail": "目录不存在"}
    ]
}
```

这个类型对动作链执行摘要特别重要。

#### progress

```python
payload = {
    "title": "正在扫描磁盘",
    "detail": "已扫描 3200 个文件",
    "current": 3200,
    "total": 10000
}
```

同时使用 `CommandResult.progress` 表示 `0.0-1.0`。

#### confirm

```python
payload = {
    "title": "确认清理",
    "detail": "将删除 35 个临时文件。",
    "danger": True,
    "confirm_action": {...}
}
```

第一阶段不建议实现 confirm。危险操作仍使用现有对话框或插件自身确认。confirm 放到后续。

### 4.3 CommandAction 升级

当前：

```python
CommandAction(type, label, value)
```

建议增加可选字段：

```python
enabled: bool = True
danger: bool = False
primary: bool = False
payload: dict[str, Any] = field(default_factory=dict)
```

为了兼容旧插件，新增字段必须有默认值。插件返回 dict 时 `core/plugin_manager.py` 的白名单也要更新。

Action 类型建议规范：

```text
copy
open_url
open_file
open_folder
show_in_explorer
save_text
save_file
create_shortcut
run_command
retry
cancel
confirm
close_qr_server
```

第一阶段必须实现或保持：

- `copy`
- `open_url`
- `open_file`
- `open_folder`
- `save_text`
- `save_file`
- `create_shortcut`
- `close_qr_server`

第二阶段再加：

- `show_in_explorer`
- `retry`
- `run_command`
- `cancel`

### 4.4 Action 数量策略

保留当前底部最多两个主按钮，但不要截断丢失动作。建议：

- `result.actions[:2]` 显示为底部按钮。
- 多于两个时显示“更多”按钮。
- 更多菜单复用 `CompactResultPopupMenu`。

因此 `limit_command_result_actions()` 不应继续永久截断。改法：

- 插件结果不再截断 action，只在 UI 渲染时限制底部按钮。
- 为避免旧 UI 溢出，`PopupCommandResultMixin._action_button_rects()` 仍只返回前两个。
- 新增 `_more_button_rect()` 和 `_show_result_more_actions_menu()`。

这是一个行为变化，需要更新测试。

## 5. 普通命令捕获输出

### 5.1 ShortcutItem 新增字段

在 `core/data_models.py` 给命令快捷方式增加：

```python
capture_output: bool = False
command_timeout_seconds: float = 10.0
command_output_max_chars: int = 20000
```

字段含义：

- `capture_output`：是否捕获 stdout/stderr 并在面板显示。
- `command_timeout_seconds`：捕获模式下的等待超时。显示窗口模式不应强制捕获。
- `command_output_max_chars`：避免输出过大卡住 UI。

默认 `capture_output=False`，保证旧命令行为不变。

### 5.2 执行行为

在 `core/shortcut_command_exec.py` 的 `_execute_command()` 中加入分支：

- `show_window=True`：保持原行为，不捕获。
- `run_as_admin=True`：第一版不捕获，因为 UAC 和降权/提权通道不适合直接拿 stdout。
- `capture_output=True` 且 `command_type in ("cmd", "python")`：用 `subprocess.Popen(... stdout=PIPE, stderr=PIPE)` 执行，等待超时后 kill。
- `command_type="builtin"`：沿用 `CommandResult` pending/result 通道。

返回结果：

```python
CommandResult(
    success=returncode == 0,
    display_type="log",
    message=combined_output,
    payload={
        "exit_code": returncode,
        "duration": duration,
        "stdout": stdout,
        "stderr": stderr,
        "command": resolved_command
    },
    actions=[
        CommandAction(type="copy", label="复制输出", value=combined_output)
    ],
    error=stderr_summary if returncode != 0 else ""
)
```

### 5.3 结果通道

当前 `ShortcutExecutor.execute()` 只能返回 `(bool, str)`。为了让普通命令结果出现在面板，有两种方案：

1. 继续使用 `set_pending_command_result()`。
2. 引入更明确的执行结果对象。

短期建议使用第一种，改动小：

```python
from core.command_registry import set_pending_command_result
set_pending_command_result(result)
return result.success, result.error
```

同时在 `PopupItemExecutionMixin.do_execute_thread()` 中，执行后尝试：

```python
pending = take_pending_command_result()
if pending:
    emit signal to UI thread -> show_command_result(pending, item.id)
```

注意：当前 `do_execute_thread()` 是后台线程，不能直接操作 Qt UI。需要新增一个 signal，例如：

```python
command_result_ready = pyqtSignal(object, str)
```

在 `LauncherPopup` 初始化时连接到 `show_command_result`。

中期应淘汰全局单槽 pending result，见第 8 节。

## 6. 结果面板 UI 增强

### 6.1 修改文件

主要修改：

```text
ui/launcher_popup/popup_command_result.py
```

可能需要联动：

```text
ui/launcher_popup/popup_window.py
ui/launcher_popup/popup_item_execution.py
tests/test_popup_search_ui.py
```

### 6.2 文本与日志

当前面板用 `QTextEdit` 显示 message。建议：

- `display_type="log"` 时使用等宽字体。
- `payload.wrap=False` 时关闭自动换行或提供水平滚动。
- 长输出截断显示时在尾部追加：

```text
[输出过长，已截断。可复制完整输出]
```

完整输出仍存在 action.value 或 payload 中，但要注意内存上限。

### 6.3 table

当前表格是手绘前 6 行，并且列宽固定 120。第一版可以继续手绘，但要支持 columns：

- 第一行绘制 columns，字体略淡。
- rows 最多绘制 6-8 行。
- 完整内容通过“复制表格” action 输出 TSV。

不要第一版引入 `QTableView`，会增加焦点、滚动、样式和遮罩复杂度。

### 6.4 kv

`kv` 可以手绘键值对：

```text
IPv4       192.168.1.8
Gateway   192.168.1.1
DNS        223.5.5.5
```

如果 item 超过 8 行，只绘制前几行，完整内容在 QTextEdit 或复制 action 中。

### 6.5 list

`list` 用于动作链摘要、检查报告。支持状态：

```text
success
failed
skipped
warning
running
```

显示符号建议使用 ASCII，避免编码问题：

```text
[OK] 打开目录
[FAIL] 执行命令: 工作目录不存在
[SKIP] 发送快捷键
```

项目当前部分文档和源码有编码显示异常，新增代码和文档尽量保持 UTF-8 且避免不必要特殊符号。

### 6.6 更多 action 菜单

当前已有 `CompactResultPopupMenu`。新增：

- `_more_button_rect()`
- `_result_button_at()` 支持 `("more", -1)`
- release 时弹菜单。

菜单内容显示 `result.actions[2:]`。执行逻辑复用 `_execute_action()`。

## 7. 参数化命令

### 7.1 现状

`CommandDefinition.params` 已存在，但 `PopupItemExecutionMixin` 执行 panel 命令时只构造：

```python
CommandContext(raw_input, args_text, clipboard_text, selected_files, update_callback)
```

`args` 没有统一填充。插件通常自己解析 `args_text`。

### 7.2 第一阶段策略

不要立刻强制所有插件改用 params。先加一个轻量解析工具：

```text
core/command_args.py
```

能力：

- 按位置参数填充 `CommandParam`。
- 支持默认值。
- 缺少 required 时返回错误。
- 不处理复杂 shell quoting，避免过度设计。后续可用 `shlex.split(..., posix=False)`。

示例：

```python
def parse_command_args(params: list[CommandParam], args_text: str) -> tuple[dict[str, str], str]:
    ...
```

在执行 panel 命令前：

```python
args, error = parse_command_args(cmd_def.params, args_text)
ctx.args = args
```

如果缺少必填参数，返回：

```python
CommandResult(
    success=False,
    message="缺少参数: host",
    error="参数错误"
)
```

### 7.3 后续输入 UI

第二阶段可以做“参数输入面板”：

- 用户输入 `/ping`，命令定义需要 `host`，弹出输入框。
- 用户输入 `/ping 8.8.8.8`，直接执行。

这与当前 `{input}` 变量收集不同。`params` 是命令协议层，`{input}` 是快捷方式模板变量层。两者要分开，不要混用。

## 8. 异步任务与取消

### 8.1 当前问题

`CommandContext.update_callback` 已存在，但没有任务管理器。插件可以调用 callback 更新 UI，但：

- 没有统一任务 ID。
- 没有取消 token。
- 没有并发管理。
- UI 关闭后更新可能打到已销毁窗口。

### 8.2 建议新增 CommandTaskManager

新增：

```text
core/command_tasks.py
```

模型：

```python
@dataclass
class CommandTask:
    id: str
    command_id: str
    cancel_event: threading.Event
    started_at: float
    latest_result: CommandResult
```

职责：

- 创建后台任务。
- 保存最近结果。
- 提供取消事件。
- 限制同时运行任务数量。
- 清理完成任务。

`CommandContext` 增加：

```python
cancel_event: threading.Event | None = None
task_id: str = ""
```

插件可检查：

```python
if context.cancel_event and context.cancel_event.is_set():
    return CommandResult(success=False, message="已取消", error="cancelled")
```

### 8.3 第一阶段不强制实现

异步任务是重要方向，但不建议和命令捕获输出同一 PR 完成。第一阶段可以先让 `update_callback` 更可靠，第二阶段再加 `CommandTaskManager`。

适合异步化的插件：

- `plugins/disk_cleaner`
- `plugins/event_inspector`
- `plugins/api_tester`
- `plugins/file_tools` 的大文件哈希
- 未来 Everything 搜索

ShortcutExecutor.run_command_capture(shortcut, timeout) -> CommandResult

## 12. 测试计划

### 12.1 核心测试

新增或扩展：

```text
tests/test_shortcut_command_exec.py
tests/test_command_registry.py
tests/test_plugin_manager.py
```

覆盖：

- `run_as_admin=True` 时禁用或跳过捕获。
- `show_window=True` 时禁用捕获。
- `CommandResult` 新字段默认兼容旧插件。

## 13. 风险与处理

### 13.1 Qt 跨线程更新

普通命令捕获输出发生在后台线程。不能直接调用 `show_command_result()`。必须用 signal 回到 UI 线程。

建议在 `LauncherPopup` 定义：

```python
command_result_ready = pyqtSignal(object, str)
```

连接：

```python
self.command_result_ready.connect(self.show_command_result)
```

后台线程：

```python
self.command_result_ready.emit(result, item.id)
```

### 13.2 全局 pending result 并发问题

当前单槽 pending result 在多命令并发时可能被覆盖。第一阶段可以接受，因为弹窗已有 `_executing` 防重入。中期必须用 task/result 对象替代。

### 13.3 输出过大

捕获输出必须限制大小。否则 `ping -t`、构建日志、大量目录输出会撑爆内存或卡 UI。

策略：

- 超时限制。
- 字符数限制。
- stderr/stdout 分开截断。
- UI 显示摘要，完整内容只保留在上限内。

### 13.4 shell 安全

不要因为捕获输出而绕过现有变量安全检查。`{clipboard}`、`{selected_text}`、`{input}` 在 CMD 中仍必须要求 `:q`。现有 `find_unquoted_external_command_variables()` 应继续在捕获分支执行。

### 13.5 编码

Windows 命令输出编码复杂。当前测试运行使用 `encoding="utf-8", errors="replace"`。真实捕获可先沿用，必要时后续增加 fallback：

- UTF-8
- mbcs
- locale preferred encoding

第一版用 `errors="replace"` 保证不崩。

## 14. 与动作链路径的关系

命令增强应先于动作链结果面板完成。原因：

- 动作链需要 `display_type=list` 展示每步摘要。
- 动作链中的命令步骤如果开启捕获输出，需要能把 stdout/stderr 纳入结果。
- 动作链未来的“上一步输出变量”依赖稳定的 `CommandResult`。

推荐依赖顺序：

```text
命令捕获输出
结果面板 list/log/table/kv
动作链 MVP
动作链执行摘要
异步任务与取消
动作链变量流转
```

## 15. 验收标准

命令增强第一轮完成后，应满足：

- 用户能创建 CMD/Python 命令并勾选“捕获输出”。
- 执行后弹窗内显示 stdout/stderr、退出码和错误摘要。
- 超时命令不会长期挂住后台线程。
- 旧命令、旧插件、旧配置不受影响。
- 插件可以返回 `table/kv/list/log` 并被稳定展示。
- 多 action 不再被模型层永久截断。
- 相关核心逻辑和 UI 绘制有测试覆盖。

## 16. 2026-05-26 修订：命令面板独立化

本计划原先以“中键弹窗结果面板增强”为主要承载，但现在产品决策调整为：

```text
中键弹窗只作为命令入口。
点击命令后，中键弹窗立即消失。
需要结果、参数输入、进度、确认或历史的命令，统一打开独立命令面板。
```

因此，本计划中涉及 `ui/launcher_popup/popup_command_result.py` 的增强内容，需要按以下原则重排：

- `popup_command_result.py` 不再作为新能力主承载，只保留兼容期或过渡期能力。
- `display_type=text/log/table/kv/list/progress/qr/confirm` 的完整渲染放到 `ui/command_panel_window.py`。
- 普通 CMD/Python 捕获输出不再“显示在弹窗结果面板”，而是显示在独立命令面板。
- `CommandDefinition.params` 驱动的参数输入应在命令面板内完成，不应在中键弹窗内弹输入框。
- 异步任务的进度、取消和完成结果由命令面板承载。
- 多 action 不需要适配中键弹窗空间限制，命令面板可以显示完整操作区。

新的实施顺序调整为：

```text
命令面板独立化
CommandResultStore
CMD/Python 捕获输出
命令面板 text/log/table/kv/list 渲染
参数化命令输入
异步进度与取消
命令历史与重新执行
```

新增和优先修改模块：

```text
core/command_results.py
ui/command_panel_window.py
ui/tray_mixins/windows_mixin.py
ui/launcher_popup/popup_item_execution.py
```

需要清理的旧路径：

- `PopupItemExecutionMixin._execute_item()` 中对 panel 命令调用 `self.show_command_result(...)` 的逻辑。
- panel 命令执行后重新 `self.show()`、`raise_()`、`activateWindow()` 的逻辑。
- `settings.enable_command_panel_v2` 这个未在 `AppSettings` 中定义的迁移期开关。

验收标准同步调整为：

- 点击需要命令面板的命令后，中键弹窗立即隐藏。
- 独立命令面板显示结果、错误、进度和 action。
- 不需要结果的命令不打开命令面板。
- 旧插件返回的 `CommandResult` 能进入独立命令面板显示。
- 中键弹窗不再承担命令结果阅读职责。

## 17. 2026-05-26 修订：命令面板尺寸模板

命令增强路径暂时不调整大的功能方向，但所有需要显示结果的增强能力都必须遵守独立命令面板的尺寸模板约束：

```text
small   接近当前中键弹窗尺寸
medium  稍大，用于普通报告
large   更大，用于日志、表格、动作链报告
auto    根据内容动态调整
```

命令或插件可以通过 `CommandResult.payload.window_size` 选择模板：

```python
CommandResult(
    display_type="log",
    payload={"window_size": "large"},
)
```

固定模板 `small/medium/large` 不允许用户手动调整大小，`auto` 才允许动态调整。四个模板中，前三个尺寸比例应尽可能一致。命令增强中的 `table/kv/list/log/progress/qr` 渲染都应优先复用 `ui/command_panel_window.py` 的模板和现有 `ThemedToolWindow` 样式，不再为不同命令各自创建独立窗口风格。

## 18. 与三条优化路径的顺序关系

命令增强不是第一优先开工项，它依赖独立命令面板先稳定。推荐顺序：

```text
命令面板独立化
命令增强
动作链
```

命令增强开始前应先具备：

- `CommandResultStore` 可以保存并读取最近结果。
- `CommandPanelWindow` 可以显示 text/log。
- `CommandPanelWindow` 支持 `small/medium/large/auto` 四种窗口模板。
- `TrayApp.show_command_panel()` 已经作为统一入口。
- `PopupItemExecutionMixin` 对 panel 命令不再调用中键弹窗内结果面板。

命令增强完成后需要为动作链提供：

- `display_type=list` 的稳定渲染。
- `display_type=log` 的输出显示。
- `CommandAction` 的完整操作区。
- `payload.window_size` 的模板选择。
- 捕获输出命令返回标准 `CommandResult`。

这样动作链可以直接复用命令增强结果协议，不需要自己做结果窗口、输出捕获和 action 按钮。
