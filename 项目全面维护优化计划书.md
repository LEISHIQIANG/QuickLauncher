# QuickLauncher 项目维护优化意见计划书

生成日期：2026-05-28  
项目目录：`G:\LEI-PLUG\QuickLauncher\QuickLauncher_V1.6.0.0`  
适用版本线：V1.6.x  
维护方向：稳定性、安全边界、诊断能力、可维护性、发布质量

---

## 1. 核心判断

QuickLauncher 当前已经不是简单 GUI 工具，而是带有系统 hook、命令执行、插件、配置恢复、自动更新、图标缓存、Windows 权限处理的桌面系统集成软件。

现阶段最需要优化的不是继续加功能，而是先把高风险路径做稳：

1. 命令执行路径要可审计、可提示、可取消、可诊断。
2. 配置读写、恢复、导入导出要可回滚、可追踪。
3. Hook、更新、插件、UI 异步任务要失败可降级。
4. 大文件要按职责拆分，避免后续修改牵一发动全身。
5. 测试要优先覆盖关键路径，而不是单纯追求覆盖率数字。

---

## 2. P0 必须优先优化的问题

P0 是会直接影响稳定性、安全性、用户数据或问题定位的事项。

| 编号 | 问题 | 优化意见 | 验收标准 |
|---|---|---|---|
| P0-01 | 命令执行入口分散，涉及 `subprocess`、`ShellExecute`、`shell=True`、管理员运行 | 建立命令执行审计表；所有入口标注是否用 shell、是否提权、是否捕获输出、是否支持超时/取消、是否需要风险提示 | 每个命令执行入口都有审计记录；高风险命令执行前有明确提示 |
| P0-02 | `shell=True` 风险高 | 逐项确认必要性；能改为参数数组的改为参数数组；不能改的写明原因和输入来源 | 不再新增无说明的 `shell=True`；已有场景都有测试或人工验收说明 |
| P0-03 | 配置损坏、恢复、修复对用户不够可见 | 配置加载、自动修复、备份恢复、导入失败都生成结构化报告，并在诊断窗口展示 | 用户能看到恢复来源、修复字段、失败原因和回滚结果 |
| P0-04 | Hook DLL 缺失或版本异常时诊断不足 | 增加 DLL 版本、能力位、hash、加载路径、最后错误码诊断 | DLL 缺失或版本不符时程序可降级启动，诊断能说明原因 |
| P0-05 | 托盘退出和后台任务清理风险 | 统一清理 hook、hotkey manager、timer、popup、update checker、后台线程 | 退出后不残留 hook；重复启动和退出稳定 |
| P0-06 | 更新失败原因不完整 | 更新检查、下载、hash 校验、安装启动、下次启动确认都写入 update state | hash 不匹配不安装；安装失败后诊断能显示上一轮更新状态 |
| P0-07 | 插件搜索源可能阻塞弹窗搜索 | 插件搜索改为异步执行，加超时和异常隔离；本地搜索优先返回 | 慢插件不会卡住弹窗输入 |
| P0-08 | 诊断包可能泄露敏感信息 | 诊断导出统一脱敏 token、password、api_key、Authorization、Cookie、URL query secret | 默认诊断包不包含常见敏感字段 |
| P0-09 | 动作链、命令超时、输出截断等常量分散 | 新增统一常量模块，避免模型、执行、健康检查规则漂移 | 自引用、互相引用、过长链判断一致 |
| P0-10 | `docs/` 被忽略但存在维护文档 | 明确正式文档是否纳入版本控制；草稿和正式文档分目录 | 架构、构建、发布、维护类正式文档可追踪 |

## 3. P1 核心模块拆分优化

P1 目标是降低维护风险。拆分时必须保持外部行为不变，不能为了拆分破坏现有功能。

### 3.1 命令执行模块

涉及文件：

- `core/shortcut_command_exec.py`
- `core/shortcut_file_exec.py`
- `core/shortcut_url_exec.py`
- `core/shortcut_chain_exec.py`
- `core/command_execution_service.py`

主要问题：

1. `shortcut_command_exec.py` 体量过大，命令构建、运行时选择、输出读取、编码、超时、取消、管理员运行、UI 回调混在一起。
2. 命令执行是全项目最高风险路径，任何小改动都可能影响安全和稳定性。
3. 健康检查、实际执行、风险提示之间存在规则漂移风险。

优化意见：

1. 新建 `core/command_exec/` 子包，拆出：
   - `runtime.py`：运行时定义。
   - `builder.py`：命令构建。
   - `process.py`：唯一子进程执行 helper。
   - `output.py`：编码识别和输出截断。
   - `risk.py`：风险识别和提示。
2. 保留旧模块作为 facade，逐步迁移调用方。
3. 统一 `CommandRunResult`，包含退出码、stdout、stderr、超时、取消、截断、编码、开始/结束时间。
4. 健康检查复用执行前 normalize 逻辑，避免检查结果和实际执行不一致。

验收标准：

- 原有命令执行测试全部通过。
- 新增覆盖：超时、取消、大输出、乱码、命令不存在、工作目录不存在、管理员运行失败。
- 所有新命令执行入口必须走统一 helper。

### 3.2 配置和数据管理模块

涉及文件：

- `core/data_manager.py`
- `core/config_validation.py`
- `core/config_repairs.py`
- `core/config_recovery.py`
- `core/config_importer.py`
- `core/config_history.py`

主要问题：

1. `data_manager.py` 同时承担配置读写、备份、恢复、导入导出、历史、图标仓库、运行状态。
2. 配置修复、验证、迁移、恢复分散，后续新增字段容易漏改。
3. `config_importer.py` 是高风险路径，但当前测试不足。

优化意见：

1. 保持 `DataManager` 对外 API 不变，内部拆成服务：
   - `ConfigStore`：读写和原子保存。
   - `ConfigBackupService`：备份和清理。
   - `ConfigRecoveryService`：损坏配置隔离和恢复报告。
   - `ConfigImportExportService`：导入导出和 dry-run。
   - `ConfigHistoryService`：历史快照。
2. 新增配置字段时必须同步修改模型、验证、修复、迁移、测试。
3. 导入前增加 dry-run 报告，导入失败必须回滚。
4. 配置修复报告应记录 old/new，不应静默改变用户配置。

验收标准：

- 坏配置可恢复，坏备份可隔离。
- 导入失败不污染当前配置。
- 用户能在诊断中看到配置恢复或修复详情。

### 3.3 内置命令和命令面板

涉及文件：

- `core/commands.py`
- `core/command_registry.py`
- `ui/command_panel_window.py`

主要问题：

1. 内置命令集中在大文件中，系统工具、设置、诊断、网络、更新混杂。
2. 命令风险等级和权限需求不够结构化。
3. 命令面板同时处理搜索、参数、执行、结果展示、焦点管理。

优化意见：

1. 内置命令按类别拆分，但保留统一注册入口。
2. 每个命令增加元数据：
   - 分类。
   - 风险等级。
   - 是否需要管理员。
   - 是否访问网络。
   - 是否修改系统。
   - 是否需要确认。
3. 命令面板拆出 search model、param form、result view、action bar。
4. 大输出结果默认截断，支持展开查看。

验收标准：

- 命令面板显示和执行行为不变。
- UI 可根据命令元数据展示风险提示。
- 插件卸载后命令和搜索源清理完整。

### 3.4 UI 设置窗口和弹窗

涉及文件：

- `ui/tray_app.py`
- `ui/tray_mixins/*`
- `ui/launcher_popup/*`
- `ui/config_window/*`

主要问题：

1. 多个 UI 文件过大，视图、数据处理、后台线程、系统调用混合。
2. QTimer、图标加载、插件搜索、网络请求可能在窗口关闭后回调。
3. UI 主流程测试薄弱，很多窗口没有 smoke test。

优化意见：

1. 所有异步任务增加 owner 或 generation id，窗口关闭后丢弃过期结果。
2. `icon_grid.py` 拆出图标模型、加载 worker、视图、批量操作。
3. `command_dialog.py` 拆出基础表单、变量面板、参数编辑、测试执行、风险预览。
4. `folder_panel.py` 的文件夹同步逻辑下沉到 core，watchdog 回调只发 signal。
5. 托盘、弹窗、设置主窗口、诊断窗口、更新窗口补 smoke test。

验收标准：

- 快速打开/关闭窗口不抛异常。
- 慢插件、慢图标、慢命令不冻结 UI。
- 退出时无后台任务访问已销毁控件。

## 4. P2 质量和发布优化

P2 不是立即重构重点，但应逐步纳入维护习惯。

| 问题 | 优化意见 | 验收标准 |
|---|---|---|
| CI 覆盖率门槛低于当前实际覆盖率 | 先提升到 35%，稳定后再考虑 40% | 覆盖率提升来自关键路径测试 |
| 编码问题可能在发布后才暴露 | CI 增加 UTF-8 解码检查 | 编码损坏能阻塞 PR |
| 版本号可能不一致 | 检查 `core/version.py`、安装器、更新元数据版本一致 | 发布前自动发现版本不一致 |
| 发布包完整性依赖人工检查 | 构建后检查 exe、hooks.dll、assets、版本、hash | 缺失关键产物时构建失败 |
| 日志和诊断分散 | 诊断包固定包含 manifest、环境、最近错误、更新状态、hook 状态、插件错误 | 用户反馈问题时能直接导出诊断包 |
| UI 测试不足 | 主窗口只先做 smoke test，不追求像素级测试 | 主流程窗口至少能构建、加载、关闭 |

## 5. 建议的实施顺序

### 第一阶段：先修风险闭环

优先处理：

1. 命令执行审计。
2. `shell=True` 审计。
3. 配置恢复报告。
4. Hook DLL 诊断。
5. 更新失败状态记录。
6. 插件搜索超时隔离。
7. 诊断包脱敏。

这一阶段的目标不是大规模重构，而是先让失败可见、风险可控。

### 第二阶段：拆核心大模块

优先处理：

1. 拆 `shortcut_command_exec.py`。
2. 拆 `data_manager.py` 内部服务。
3. 拆 `commands.py` 命令类别。
4. 拆 `command_dialog.py` 和 `icon_grid.py` 的重逻辑。

这一阶段必须小步提交，每次拆分后保持测试通过。

### 第三阶段：补测试和发布门禁

优先处理：

1. `config_importer.py` 安全测试。
2. 更新 UI smoke test。
3. 诊断窗口 smoke test。
4. 托盘生命周期测试。
5. CI 增加编码、版本、产物完整性检查。
6. 覆盖率门槛提升到 35%。

---

## 6. 暂不建议做的事

这些事项短期投入大、收益不够直接，建议暂缓：

1. 暂不迁移 PyQt6。
2. 暂不支持 macOS/Linux。
3. 暂不做插件进程级强沙箱。
4. 暂不重写安装器体系。
5. 暂不一次性清理全部 `except Exception`。
6. 暂不为了覆盖率数字写无意义测试。
7. 暂不重写全部 UI。

---

## 7. 最终优化结论

当前最合理的维护路线是：

1. 先把命令执行、配置恢复、更新、hook、插件这些高风险路径补齐诊断和失败处理。
2. 再拆最大、最危险的核心模块，拆分过程中保持兼容 facade，避免一次性破坏调用方。
3. UI 优化先关注生命周期和异步安全，不优先做视觉重写。
4. 测试优先覆盖配置导入、命令执行、更新、hook、托盘和主窗口，不追求表面覆盖率。
5. 发布流程要加入版本、hash、产物完整性和诊断包检查，减少人工漏项。

完成以上优化后，项目才适合继续扩展插件能力、命令系统和更多自动化功能。
