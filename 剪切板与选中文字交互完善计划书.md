# QuickLauncher 剪切板与选中文字交互完善计划书

## 1. 背景与目标

QuickLauncher 目前已经具备两类 Windows 上下文输入能力：

1. 剪切板文本：用于命令变量、内置命令兜底输入、插件 API、搜索框粘贴、结果复制。
2. 资源管理器选中文件：通过 Explorer COM 获取选中文件，并绑定到触发窗口上下文。

但从代码结构看，剪切板和选中文字还没有形成与“选中文件”同等级的系统能力。剪切板逻辑分散在命令变量、命令执行、插件管理、弹窗 UI 中；选中文字逻辑只作为 `{{selected_text}}` 的临时实现存在，依赖“窗口淡出后运行 + 模拟 Ctrl+C + 恢复剪切板”，没有独立服务、没有多策略读取、没有统一的上下文模型。

本计划的目标是把“剪切板”和“选中文字”从零散工具能力升级为 QuickLauncher 的核心 Windows 交互能力：

- 建立统一的 `ClipboardService`，提供可靠读写、格式识别、快照恢复、历史监听。
- 建立统一的 `SelectedTextService`，提供多策略读取选中文字，不再只依赖命令变量内部逻辑。
- 建立统一的 `InteractionContext`，让弹窗、命令、插件、动作推荐共享同一份上下文。
- 强化用户体验：打开面板即可基于剪切板/选中文字推荐动作。
- 强化安全与隐私：避免误读密码、敏感窗口、私密剪切板内容。
- 强化测试：覆盖文本、文件、图片、HTML、失败恢复、并发和权限边界。

## 2. 当前实现梳理

### 2.1 剪切板读取路径

当前主要读取路径如下：

| 位置 | 当前职责 | 问题 |
| --- | --- | --- |
| `core/command_variables.py::read_clipboard_text()` | 使用 `win32clipboard` 读取 `CF_UNICODETEXT` / `CF_TEXT` | 只支持纯文本；没有重试；没有格式识别；异常静默吞掉 |
| `core/shortcut_command_exec.py::_capture_selected_text()` | 为选中文字临时快照剪切板、模拟复制、恢复剪切板 | 与选中文字强耦合；快照格式有限；没有互斥锁 |
| `core/shortcut_command_exec.py::_snapshot_clipboard()` | 枚举部分剪切板格式 | 只支持 CF_UNICODETEXT / CF_TEXT / CF_HDROP 三种格式；保存/恢复复杂格式可能失败；不适合复用 |
| `core/plugin_manager.py::read_clipboard()` | 插件 API 读取 Qt 剪切板文本 | 与命令变量读取方式不一致；只读文本 |
| `core/plugin_manager.py::write_clipboard()` | 插件 API 写入 Qt 剪切板文本 | 只支持文本；没有 HTML/文件/图片 |
| `ui/launcher_popup/popup_search.py::_read_clipboard_text()` | 弹窗搜索框右键粘贴 | 使用 Qt 剪切板；与核心逻辑分离 |
| `ui/launcher_popup/popup_item_execution.py` | 执行面板命令前抓取 `clipboard_text` | 只抓文本快照；没有类型信息 |
| `core/shortcut_url_exec.py` | URL 变量里读取 `{{clipboard}}` | 有独立的 _TOKEN_RE 正则和 _resolve_url_variables() 解析器，与 command_variables 是两套独立变量系统 |
| `core/command_registry.py::CommandContext` | 传递剪切板文本到内置命令 | 只有 clipboard_text 字段，没有 clipboard_kind / clipboard_files / selected_text |

额外风险：plugin_manager.read_clipboard() 使用 QApplication.clipboard()，只能在 Qt 主线程调用；插件后台线程调用时可能段错误。command_variables.read_clipboard_text() 使用 win32clipboard，在 Qt 主线程（默认 MTA）调用需要 STA COM 初始化，当前能工作是巧合而非设计。

结论：剪切板目前是“纯文本工具函数”，不是“上下文服务”。

### 2.2 选中文字读取路径

当前主要逻辑：

1. 弹窗显示前调用 `ShortcutExecutor.save_foreground_window()` 保存原前台窗口。
2. 命令变量解析时，如果命令快捷项 `trigger_mode == "after_close"`，给 `{{selected_text}}` 注入 `_capture_selected_text`。
3. `_capture_selected_text()` 尝试恢复之前的前台窗口。
4. 通过 `_execute_hotkey_sendinput(["ctrl"], "c")` 模拟复制。
5. 轮询 `GetClipboardSequenceNumber()` 判断剪切板是否变化。
6. 读取剪切板文本。
7. 恢复之前的剪切板快照。

当前限制：

- 只有命令变量使用 `{{selected_text}}` 时才触发。
- 只在 `after_close` 模式下可用；立即执行会报错或被 UI 自动改成 after_close。
- 没有 UI Automation、Win32 标准控件消息等非侵入式读取策略。
- 没有识别目标窗口、进程、控件类型。
- 没有向插件和内置命令暴露 `selected_text`。
- 没有选中文字状态提示或动作推荐。
- 失败时通常只得到空字符串，用户难以理解失败原因。

### 2.3 选中文件能力作为参考

`ui/launcher_popup/file_selection.py` 已经有较成熟的设计：

- `SelectionTriggerContext` 保存触发方式、鼠标位置、前台窗口、根窗口、目标类型。
- 通过 Explorer COM 获取 `SelectedItems()`。
- 异步检测，避免阻塞 UI。
- 绑定触发窗口，避免把错误窗口的选择误认为当前输入。
- 弹窗中缓存 `_selected_files`、`_selected_files_status`、来源 hwnd、捕获时间。

剪切板和选中文字应参考这个设计，形成类似的上下文快照能力。

## 3. 核心问题

### 3.0 分类匹配策略（核心设计决策）

分类器不仅要定义"有哪些类型"，还要定义"当一条内容同时匹配多个类型时，如何决策"：

**匹配优先级规则（按优先顺序）**：

1. **明确拒绝型 > 结构签名型 > 内容匹配型 > 模糊识别型**
   - empty（空）→ JWT/JSON（有明确结构签名）→ URL/Email/IP（正则内容匹配）→ path/color/code（模糊启发式）
2. **置信度门槛**：默认 `threshold >= 0.8` 才采用该分类结果；低于门槛则降级为 `unknown`
3. **同级别冲突**：同级别多个匹配时，取置信度最高的；置信度相同时取字符串匹配长度最短的（更具体的模式优先）
4. **显示优先级独立于分类优先级**：显示优先级仅决定 §10.1 中的图标展示顺序（URL > JSON > 文件 > 图片 > 文本），不影响分类结果本身
5. **是否支持 "多标签"**：第一阶段仅返回单一 `kind`（最确定的一个）；第二阶段支持返回 `kind` + `secondary_kinds: list[str]`，供 UI 展示更多可能的动作

**变更记录**：此规则会在 Phase 3 分类器实现时正式落地，但在 Phase 1 的数据模型中就需要预留 `confidence` 字段。

### 3.1 结构问题

- 剪切板读写散落在多个模块，没有统一入口。
- 选中文字逻辑藏在 `ShortcutExecutor` 内部，无法被内置命令、插件、动作推荐复用。
- `CommandContext` 只有 `clipboard_text` 和 `selected_files`，没有 `selected_text`、`clipboard_snapshot`、`clipboard_kind`。
- 插件 API 权限只有 `clipboard.read` / `clipboard.write`，表达能力不足。

### 3.2 能力问题

- 只支持纯文本，未充分利用 Windows 剪切板多格式能力。
- 不支持文件列表、图片、HTML、RTF 的结构化读取。
- 不支持剪切板历史、固定片段、搜索。
- 不支持基于剪切板类型的动作推荐。
- 选中文字没有多策略读取，稳定性依赖目标应用对 Ctrl+C 的响应。

### 3.3 稳定性问题

- `OpenClipboard()` 可能被其他进程占用，当前多数实现没有重试。
- 选中文字读取过程中可能误覆盖剪切板。
- 快照恢复没有统一策略，对复杂格式支持不明确。
- 没有全局剪切板互斥锁，多个路径并发读写时可能互相干扰。
- 高权限/低权限窗口之间可能存在 UIPI 限制。
- 线程模型冲突：Qt 主线程默认 MTA COM，win32clipboard 需要 STA，当前无统一 COM 初始化策略。
- Qt 剪切板与 Win32 剪切板不同步：QApplication.clipboard() 有内部缓存，与 GetClipboardData() 返回值可能不一致。
- shortcut_url_exec.py 有独立的变量解析系统（自己的 _TOKEN_RE），ClipboardService 统一后需同步替换两处。

### 3.4 用户体验问题

- 用户不容易知道当前是否检测到了剪切板内容、选中文字、文件选择。
- `{{selected_text}}` 是隐藏能力，配置成本高。
- 剪切板只是命令输入兜底，不会主动变成快捷动作。
- 失败原因不透明，例如目标窗口没有选中文字、复制超时、剪切板被占用。

### 3.5 安全与隐私问题

- 剪切板可能包含密码、Token、验证码、私密内容。
- 如果做历史记录，需要明确排除规则、脱敏策略、最大容量和清除入口。
- 插件读取剪切板时应保留权限边界，并可扩展到更细粒度权限。
- 选中文字读取会模拟复制，可能触发目标应用副作用。
- 高权限窗口（管理员、UAC 弹窗、锁屏界面）的 SendInput/AttachThreadInput 操作会被 UIPI 拦截，需提前检测。

## 4. 目标架构

### 4.1 总体模型

InteractionContext 在弹窗打开时生成快照，之后只传递快照数据，不直接持有服务引用：

```text
弹窗触发
  │
  ├── 同步 (T+0ms) ──→ TriggerContext + 轻量剪切板文本
  │
  ├── 异步 (T+10ms) ─→ ClipboardSnapshot + ClipboardClassification
  │                        SelectedFilesContext (已有逻辑)
  │                        SelectedTextResult (仅非侵入式: UIA/Win32 Edit)
  │
  └── 命令执行时 ──→ CommandContext (clipboard_text + clipboard_kind + selected_text ...)
```

数据流：

```text
ClipboardService ──→ ClipboardSnapshot ──→ ClipboardClassification
                                                    │
SelectedTextService ──→ SelectedTextResult           │
                                                    ▼
                              InteractionContext ──→ Action Suggestions
                              (弹窗打开时快照)        CommandContext (命令执行时)
                                                    PluginContext (插件执行时)
```

其中：

- `TriggerContext`：一次弹窗/快捷键触发时的窗口、鼠标、时间、进程上下文。
- `ClipboardContext`：当前剪切板内容、格式、分类、摘要。**注意：InteractionContext 不直接持有 ClipboardService 引用，只持有打开时的快照数据。**
- `SelectedTextContext`：当前前台窗口选中文字、读取方式、失败原因。
- `SelectedFilesContext`：已有的资源管理器选中文件上下文。
- `InteractionContext`：统一上下文，给 UI、命令、插件、动作推荐使用。

### 4.2 新增核心模块

建议新增：


## 5. ClipboardService 设计

### 5.1 基础 API

#### COM 初始化策略（关键阻塞问题）

`win32clipboard.OpenClipboard()` 隐式要求调用线程已初始化 STA（Single-Threaded Apartment）COM。当前代码"能工作"是巧合而非设计——Python 线程默认未初始化 COM，`win32clipboard` 模块在首次调用时会自动 `CoInitializeEx(COINIT_APARTMENTTHREADED)`，但这在某些环境下可能失败（如 Qt 主线程已初始化为 MTA 时）。

**强制策略**：

```python
class Win32ClipboardImpl:
    _com_local = threading.local()
    
    def _ensure_sta(self):
        """Ensure STA COM on current thread before clipboard operations."""
        if not getattr(self._com_local, '_com_initialized', False):
            import pythoncom
            try:
                pythoncom.CoInitializeEx(pythoncom.COINIT_APARTMENTTHREADED)
            except pythoncom.CO_E_ALREADYINITIALIZED:
                pass  # Already initialized (may be MTA — proceed with caution)
            except Exception:
                # Cannot initialize COM — use dedicated STA worker thread
                self._use_worker_thread = True
            self._com_local._com_initialized = True
    
    @property
    def _needs_worker(self) -> bool:
        return getattr(self, '_use_worker_thread', False)
```

如果当前线程无法初始化为 STA（例如 Qt 主线程已在 MTA 模式），`ClipboardService` 自动 fallback 到专用 STA 工作线程：

```python
def _run_on_sta_thread(self, func, *args, **kwargs):
    """Run clipboard operation on a dedicated STA thread."""
    if not hasattr(self.__class__, '_sta_thread'):
        from queue import Queue
        result_queue = Queue()
        def _sta_worker():
            pythoncom.CoInitializeEx(pythoncom.COINIT_APARTMENTTHREADED)
            while True:
                cmd, args, kwargs, q = queue.get()
                try:
                    q.put(func(*args, **kwargs))
                except Exception as e:
                    q.put(e)
        t = threading.Thread(target=_sta_worker, daemon=True, name="ClipboardSTA")
        t.start()
        self.__class__._sta_thread = t
        self.__class__._sta_queue = queue.Queue()
    # Post to STA thread and wait for result
    ...
```

#### 锁策略

使用**模块级全局 RLock**（跨所有 Win32 操作），而不是每实例锁：

```python
_clipboard_lock = threading.RLock()

class Win32ClipboardImpl:
    def open_clipboard(self, timeout_ms=200):
        with _clipboard_lock:
            # 8-round exponential backoff OpenClipboard
            ...
```

原因：
- `OpenClipboard()` 是进程级锁，不是线程级锁——多线程同时调用只有一个能成功，需要排队
- 全局 RLock 确保同一进程内不会出现两个线程同时操作 `OpenClipboard()`
- Qt 实现（`QtClipboardImpl`）**不需要锁**，它只在主线程调用，且 Qt 内部有自己的锁

#### QtClipboardBridge

当后台线程通过 Win32 路径写入剪切板后，Qt 的 `QClipboard.dataChanged` 信号**不会自动触发**，依赖此信号的插件会读到旧值。

```python
class QtClipboardBridge(QObject):
    """Bridge Win32 clipboard writes to Qt clipboard signal."""
    
    _signal = pyqtSignal()
    
    def __init__(self):
        super().__init__()
        self._signal.connect(self._sync_qt_clipboard)
    
    def notify_change(self):
        """Call from any thread after Win32 clipboard write."""
        self._signal.emit()  # Queued connection → main thread
    
    def _sync_qt_clipboard(self):
        """Main thread: sync QClipboard from Win32."""
        if not QApplication.instance():
            return
        text = self._win32_read_text()  # Re-read via Win32
        if text is not None:
            QApplication.clipboard().setText(text)
```

激活条件：仅在 `write_text()` / `write_html()` 由非主线程调用且 Qt 主线程可用时触发。

建议接口：

```python
class ClipboardService:
    def read_text(self, timeout_ms: int = 200) -> str: ...
    def write_text(self, text: str, *, preserve_history: bool = True) -> bool: ...
    def read_snapshot(self, timeout_ms: int = 200) -> ClipboardSnapshot: ...
    def restore_snapshot(self, snapshot: ClipboardSnapshot, timeout_ms: int = 300) -> bool: ...
    def get_sequence_number(self) -> int: ...
    def get_available_formats(self) -> list[ClipboardFormatInfo]: ...
    def is_format_available(self, format_id: int) -> bool: ...
    def classify(self, snapshot: ClipboardSnapshot | None = None) -> ClipboardClassification: ...
    def register_listener(self, callback: Callable[[int], None]) -> None: ...  # 序号变化通知
    def unregister_listener(self, callback) -> None: ...
    def get_stats(self) -> dict: ...  # 重试次数、命中率等运维信息
```

**线程模型**：ClipboardService 内部维护两种实现——Win32Impl（线程安全，非主线程使用）和 QtImpl（触发 Qt dataChanged 信号，仅主线程使用）。实例化时自动检测调用线程：

```python
class ClipboardService:
    def _get_impl(self) -> IClipboardImpl:
        if threading.current_thread() is threading.main_thread():
            return self._qt_impl
        return self._win32_impl
```

### 5.2 数据模型

```python
@dataclass
class ClipboardFormatInfo:
    format_id: int
    name: str
    readable: bool
    size_hint: int = 0

@dataclass
class ClipboardSnapshot:
    sequence: int
    captured_at: float
    formats: dict[int, object]  # 仅存储 "可序列化友好" 格式的二进制数据（见下文）
    text: str = ""
    file_paths: list[str] = field(default_factory=list)
    html: str = ""
    rtf: bytes = b""
    image_info: dict = field(default_factory=dict)
    source: str = "win32"
    truncated: bool = False
    error: str = ""

    @property
    def has_image(self) -> bool:
        return bool(self.image_info and "width" in self.image_info)

    @property
    def is_empty(self) -> bool:
        return not self.text and not self.file_paths and not self.html and not self.image_info

#### 快照格式存储策略

`formats` 字典不是"所有格式的完整二进制副本"，而是按策略有选择地存储：

- **存储二进制数据**：`CF_UNICODETEXT`, `CF_TEXT`, `CF_HDROP`
- **存储结构化的**：`HTML Format` 存纯文本, `RTF` 存 bytes
- **只检测不存储**：`CF_DIB`, `CF_BITMAP`, `CF_DIBV5`, `CF_METAFILEPICT`, 自定义私有格式
  - 记录 `format_id`, `name`, `size_hint` 但不存储二进制内容
  - 图片仅记录 `image_info`（宽/高/格式名/大致字节大小）
- **明确的排除列表**：`CF_LOCALE`, `CF_OEMTEXT`, `CF_PALETTE`, 以及所有 > 10MB 的自定义格式

此策略在 `read_snapshot()` 中强制执行，调用者无法绕过。

@dataclass
class ClipboardClassification:
    kind: str
    confidence: float
    summary: str
    actions: list[str] = field(default_factory=list)
    metadata: dict = field(default_factory=dict)
```

### 5.3 剪切板打开策略

必须统一封装 `OpenClipboard()`：

- 使用进程内全局 `RLock` + 指数退避重试。
- 遇到占用时重试 8 次，间隔 [10, 20, 50, 100, 100, 100, 200, 200]ms。
- 首次失败时调用 `GetOpenClipboardWindow()` 获取占用窗口句柄并记录诊断信息。
- `OpenClipboard()` 是进程级锁（非线程级），不同进程间只有一个能打开，重试是必需的。
- 记录失败原因：`open_timeout`、`format_unreadable`、`restore_failed`。
- 所有调用者禁止直接使用 `win32clipboard.OpenClipboard()`。
- 重试失败后抛出 `ClipboardOpenError` 而非静默忽略。

### 5.4 格式支持优先级

第一阶段（读写+快照）：

- `CF_UNICODETEXT` （读写）
- `CF_TEXT` （读写 fallback）
- `CF_HDROP` （只读，文件列表）
- `HTML Format` （只读，富文本）
- `CF_DIB` / `CF_BITMAP` / `CF_DIBV5` **仅做存在性检测和尺寸摘要读取**，不做完整图片二进制反序列化。原因：CF_BITMAP 是 GDI 句柄跨进程无效，CF_DIB/DIBV5 反序列化消耗大。

第二阶段：

- RTF 格式读取
- PNG / JFIF 等自定义格式检测

### 5.7 错误类型体系

统一错误类型，替代当前以空字符串 + 静默异常为主的错误处理方式：

```python
class ClipboardError(RuntimeError):
    """Base for all clipboard errors."""
    def __init__(self, code: str, message: str = "", detail: dict = None):
        self.code = code        # 机器可读的错误码
        self.detail = detail or {}
        super().__init__(message or code)

class ClipboardOpenError(ClipboardError):
    """OpenClipboard 重试耗尽后抛出"""
    code = "open_timeout"

class ClipboardFormatUnreadableError(ClipboardError):
    """指定格式无法读取（被占用/权限不足/格式不支持）"""
    code = "format_unreadable"

class ClipboardEmptyError(ClipboardError):
    """剪切板为空"""
    code = "empty"

class ClipboardRestoreError(ClipboardError):
    """快照恢复失败"""
    code = "restore_failed"

class ClipboardComError(ClipboardError):
    """COM 初始化失败"""
    code = "com_init_failed"

class ClipboardUipiError(ClipboardError):
    """UIPI 阻止跨权限读取"""
    code = "uipi_blocked"
```

**使用规则**：
- `read_text()` 失败返回空字符串（不抛异常——大多数调用者不需要中断流程）

- `open_clipboard()` 失败抛出 `ClipboardOpenError`（底层重试耗尽后，不应静默）

### 6.3 Win32 Edit/RichEdit 策略

针对传统控件：

- `GetForegroundWindow()`
- `GetFocus()` + `AttachThreadInput()`
- 判断类名是否为 `Edit`、`RichEdit20W`、`RICHEDIT50W` 等。
- `EM_GETSEL` 获取选区。
- `WM_GETTEXTLENGTH` / `WM_GETTEXT` 获取文本。
- 根据选区截取。

**AttachThreadInput 超时机制（关键安全措施）**：

`AttachThreadInput()` 本身没有超时能力——如果目标线程死锁，它可能永久阻塞。必须通过预检查防止：

```python
def _try_win32_edit(self, hwnd: int, timeout_ms: int = 200) -> str | None:
    thread_id = ctypes.windll.user32.GetWindowThreadProcessId(hwnd, None)
    if not thread_id:
        return None
    
    # 1. 等待目标线程空闲（最多 timeout_ms）
    kernel32 = ctypes.windll.kernel32
    target_process = None
    try:
        process_id = ctypes.wintypes.DWORD()
        ctypes.windll.user32.GetWindowThreadProcessId(hwnd, ctypes.byref(process_id))
        target_process = kernel32.OpenProcess(0x0400, False, process_id.value)  # SYNCHRONIZE
        if target_process:
            # WaitForInputIdle: 等待线程进入空闲状态
            WAIT_TIMEOUT = 0x102
            result = kernel32.WaitForInputIdle(target_process, timeout_ms)
            if result == WAIT_TIMEOUT:
                return None  # 目标线程无响应 → 跳过此策略
    finally:
        if target_process:
            kernel32.CloseHandle(target_process)
    
    # 2. AttachThreadInput（必须在 finally 中 detach）
    attached = False
    try:
        current_id = kernel32.GetCurrentThreadId()
        attached = bool(ctypes.windll.user32.AttachThreadInput(current_id, thread_id, True))
        if not attached:
            return None
        # ... EM_GETSEL / WM_GETTEXT ...
    finally:
        if attached:
            ctypes.windll.user32.AttachThreadInput(current_id, thread_id, False)
```

**`WM_GETTEXT` 使用 `SendMessageTimeout`**：

```python
def _send_gettext(hwnd: int, timeout_ms: int = 200) -> str | None:
    """使用 SendMessageTimeout 发送 WM_GETTEXT，避免死锁"""
    SMTO_ABORTIFHUNG = 0x0002
    WM_GETTEXTLENGTH = 0x000E
    WM_GETTEXT = 0x000D
    
    length = ctypes.windll.user32.SendMessageTimeoutW(
        hwnd, WM_GETTEXTLENGTH, 0, 0,
        SMTO_ABORTIFHUNG, timeout_ms
    )
    if not length or length <= 0:
        return None
    buf = ctypes.create_unicode_buffer(length + 1)
    result = ctypes.windll.user32.SendMessageTimeoutW(
        hwnd, WM_GETTEXT, length + 1, buf,
        SMTO_ABORTIFHUNG, timeout_ms
    )
    return buf.value if result else None
```

**`EM_GETSEL` 的 32/64 位兼容**（见下方 §6.6 详细说明）：

`EM_GETSEL` 的 `wParam` / `lParam` 在 32 位和 64 位进程间行为不同，需通过返回值获取选区而非传入指针。

**超时汇总**：Win32 Edit 策略的总超时上限为 `200ms`（`WaitForInputIdle` 100ms + `SendMessageTimeout` 各 50ms × 2 次）。

优点：

- 快、稳定、不改剪切板。

限制：

- 只适合标准控件。
- 对浏览器、VS Code、JetBrains、微信等基本无效。

### 6.4 Clipboard fallback 策略

保留现有方案，但服务化并增强：

1. 记录原剪切板 `sequence_number`。
2. 使用 `ClipboardService.read_snapshot()` 保存快照。
3. 恢复触发前台窗口。
4. 等待目标窗口激活。
5. 发送 `Ctrl+C`。
6. 轮询剪切板序号变化。
7. 读取文本。
8. 如果文本为空但剪切板变成文件或图片，不认为是有效选中文字。
9. 恢复快照。
10. 返回详细结果。

增强点：

- 增加互斥锁，避免多个复制操作并发。
- 对 QuickLauncher 自己的窗口直接拒绝，避免复制搜索框内容。
- 允许配置最大等待时间。
- 允许在失败时不覆盖用户剪切板。
- 记录 `method="clipboard_copy"`、`error="timeout/no_text/restore_failed"`。

### 6.6 EM_GETSEL 的 32/64 位进程兼容性

`EM_GETSEL` 在跨 32/64 位进程使用时需要特别注意：

**问题**：当 QuickLauncher 是 64 位进程，目标窗口是 32 位进程（或反过来）时，`SendMessage(hwnd, EM_GETSEL, &wParam, &lParam)` 传入的指针被截断或解释错误。

**方案**：不使用 `wParam` / `lParam` 传出参数，而是通过返回值获取选区：

```python
def _get_selection_range(hwnd: int) -> tuple[int, int] | None:
    """通过 EM_GETSEL 返回值获取选区，兼容 32/64 位跨进程。"""
    EM_GETSEL = 0x00B0
    result = ctypes.windll.user32.SendMessageW(hwnd, EM_GETSEL, 0, 0)
    # 返回值 LOBYTE(LOWORD) = start, HIBYTE(HIWORD) = end
    # 在 64 位进程中：lResult 的低 32 位 = start << 16 | end
    sel_start = result & 0xFFFF
    sel_end = (result >> 16) & 0xFFFF
    if sel_start == 0 and sel_end == 0:
        return None  # 无选区
    return (sel_start, sel_end)
```

注意：对于超过 65535 字符的文本，`EM_GETSEL` 返回值会被截断。对于这种情况，如果 `WM_GETTEXTLENGTH` 返回长度 > 65535，则需要使用 `EM_GETSEL` 的指针参数版本（通过 VirtualAllocEx 在目标进程中分配内存），但第一阶段不处理这种极端情况，超过 65535 字符时跳过 Win32 Edit 策略，fallback 到 Clipboard fallback。

## 11. 剪切板历史设计

### 11.1 监听方式

Windows 推荐：

- `AddClipboardFormatListener(hwnd)`
- 处理 `WM_CLIPBOARDUPDATE`
- 读取剪切板摘要
- 通过队列异步处理，避免阻塞消息循环

Qt 中可选：

- 用隐藏 native window 接收消息。
- 或使用 Qt clipboard `dataChanged` 作为跨平台兜底。

建议 Windows 优先使用 `AddClipboardFormatListener`，Qt 作为 fallback。

### 11.2 存储

建议直接使用 SQLite（sqlite3 是 Python 标准库，零额外依赖），比 JSONL 更可靠：

- 支持原子写入、并发读
- 高效的时间/类型查询和去重
- 支持快速删除和清空

```sql
CREATE TABLE clipboard_history (
    id TEXT PRIMARY KEY,
    kind TEXT NOT NULL,          -- 'text','url','json','file_list','image','html'
    text_content TEXT,           -- 纯文本内容（图片/文件时为 NULL）
    file_paths TEXT,             -- JSON 数组或 NULL
    html_content TEXT,           -- HTML 内容或 NULL
    image_hash TEXT,             -- 图片 SHA256（不存原图）
    image_width INTEGER,         -- 图片宽度
    image_height INTEGER,        -- 图片高度
    source_process TEXT,         -- 来源进程名
    content_hash TEXT NOT NULL,  -- 去重用 SHA256
    created_at REAL NOT NULL,    -- time.time()
    pinned INTEGER DEFAULT 0,   -- 0/1 固定
    size_hint INTEGER DEFAULT 0 -- 内容字节数
);
CREATE INDEX idx_history_created ON clipboard_history(created_at DESC);
CREATE INDEX idx_history_kind ON clipboard_history(kind);
CREATE UNIQUE INDEX idx_history_hash ON clipboard_history(content_hash);
```

如果后续需要多进程访问（如多个 QL 实例），可考虑 JSONL 或锁文件机制兜底。

**Schema 迁移策略**：

```sql
-- 每次打开数据库时检查 PRAGMA user_version
PRAGMA user_version;

-- Schema v1（初始版本）
CREATE TABLE IF NOT EXISTS clipboard_history (...);
PRAGMA user_version = 1;

-- 后续迁移示例（v1 → v2）：
-- if user_version == 1:
--     ALTER TABLE clipboard_history ADD COLUMN source_app TEXT;
--     PRAGMA user_version = 2;
```

迁移规则：
- 使用 Python 侧迁移检查（在 `ClipboardHistory.__init__()` 中执行），而非 SQLite 触发器
- 每次升级递增 `user_version`
- 每个迁移步骤必须是等幂的（`CREATE IF NOT EXISTS` / `ALTER ... IF NOT EXISTS` 安全）
- 迁移失败时：日志记录失败信息，以只读模式打开数据库，不阻塞应用启动
- 数据文件路径：`{config_dir}/clipboard_history.db`

### 11.3 去重与限制

- 按内容 hash 去重。
- 连续相同内容不重复记录。
- 文本最大保存长度默认 100 KB。
- 图片默认不保存原图，只保存摘要；后续可选保存缩略图。
- 文件列表保存路径，但应支持隐私开关。
- 默认历史条数 100。

### 11.4 隐私过滤

应默认过滤：

- 密码管理器窗口（窗口标题包含 "KeePass"、"Bitwarden"、"1Password" 等）。
- 浏览器隐身窗口标题（包含 "InPrivate"、"Incognito" 等）。
- 远程桌面窗口（标题包含 "远程桌面"、"Remote Desktop"）。
- 疑似密钥的文本（使用精确正则而非前缀匹配，避免误杀）：

| 模式 | 正则 | 误判风险 |
|------|------|---------|
| OpenAI API Key | `sk-[a-zA-Z0-9]{20,}` | 低 |
| GitHub Token | `ghp_[a-zA-Z0-9]{36}` | 低 |
| AWS Access Key | `AKIA[0-9A-Z]{16}` | 低 |
| JWT Token | `^eyJ[a-zA-Z0-9_-]+\.eyJ[a-zA-Z0-9_-]+\.[a-zA-Z0-9_-]+$` | 中 |
| SSH 私钥 | `-----BEGIN.*PRIVATE KEY-----` | 低 |

- 过短验证码是否保存需要配置，默认不固定。

过滤策略：

- `do_not_store`
- `store_redacted`
- `store_metadata_only`

## 12. 安全设计

### 12.1 剪切板安全

- 所有外部输入进入命令前继续执行 `:q` 检查。
- 历史功能默认关闭或首次启用时明确提示。
- 插件读取历史需要高风险权限。
- 不把剪切板内容写入普通日志。
- Debug 日志记录策略：

```python
# ✅ 记录格式：类型 + 长度 + hash 前缀（用于调试追踪）
logger.debug("clipboard: %d bytes, type=%s, hash=%s",
             len(text), kind, hashlib.sha256(text.encode())[:8].hexdigest())

# ❌ 禁止
logger.info("clipboard content: %s", text)
print(text)
```

- 日志中禁止包含剪切板全文，禁止 print() 输出剪切板内容。

### 12.2 选中文字安全

- 模拟 Ctrl+C 前确认目标窗口不是 QuickLauncher 自己。
- 读取高权限窗口失败时返回明确错误，不尝试绕过权限。
- 目标窗口变化时放弃读取，避免读取错误应用。
- 可以配置应用黑名单。
- **UAC 高权限窗口检测**：通过 GetWindowThreadProcessId → OpenProcess → 检查进程完整性级别。非管理员进程向管理员窗口模拟输入会被 UIPI 拦截。
- **锁屏界面检测**：前景窗口为 LogonUI.exe 时直接拒绝。
- **全屏游戏检测**：检测前景窗口是否为全屏 DirectX/OpenGL 窗口，模拟输入可能被 Hook 拦截。
- **终端窗口检测**：Windows 终端中 Ctrl+C 是 SIGINT 信号不是复制，需跳过 clipboard fallback。

### 12.3 插件安全

- 插件不得直接获得完整 `ClipboardSnapshot`，除非声明对应权限。
- 默认 `clipboard.read` 只给纯文本。
- `clipboard.history.read` 必须标为高风险，并在安装/启用时提示。

## 15. 兼容性策略

- 保持 `{{clipboard}}`、`{{clipboard:q}}` 语义不变。
- 保持 `{{selected_text}}` 需要窗口淡出模式的初期规则。
- 保持 `PluginAPI.read_clipboard()` 返回字符串（内部实现可切换）。
- 保持 `CommandContext.clipboard_text` 字段。
- 所有新字段默认空值。
- 先服务化再增强 UI，避免大范围回归。
- 旧函数替换后保留 2 个小版本的兼容包装器，标记 `@deprecated`。
- 在日志中记录被废弃的调用路径，便于统计残留使用。
- 所有新设置项使用 `AppSettings` 的 `get()` 带默认值访问，不修改已有键值。

## 16. 风险与应对

### 16.1 UI Automation 阻塞

风险：某些应用 UIA 调用很慢。

应对：

- 放到后台线程。
- 短超时。
- 默认失败后快速 fallback。

### 16.2 剪切板恢复失败

风险：复杂格式无法恢复，用户剪切板被改变。

应对：

- 第一阶段优先完整恢复文本和文件列表。
- 复杂格式恢复失败时至少不写空。
- 返回 `restore_failed` 并记录摘要日志。

### 16.3 历史隐私风险

风险：保存密码、Token、隐私内容。

应对：

- 历史默认关闭或首次启用提示。
- 敏感规则默认开启。
- 提供一键清空。
- 不保存来自黑名单应用的内容。

### 16.4 动作推荐噪音

风险：打开弹窗出现过多无关动作。

应对：

- 只在搜索为空时显示。
- 每类上下文最多 2-3 个动作。
- 用户可关闭上下文推荐。

### 16.5 线程模型冲突（新增）

风险：ClipboardService 在 Qt 主线程（默认 MTA）和后端子线程（未初始化 COM）之间切换使用时，win32clipboard 行为不一致。

应对：

- ClipboardService 内部使用双实现策略（详见 §5.1）：Win32 实现自动初始化 STA COM（`CoInitializeEx(COINIT_APARTMENTTHREADED)`），在主线程无法初始化为 STA 时切换到专用 STA 工作线程。
- Qt 主线程路径保留 `QApplication.clipboard()` 实现（`QtClipboardImpl`），不走 Win32。
- 写入一致性通过 `QtClipboardBridge` 保证：后台线程写入后通知主线程同步 QClipboard。
- 关键验证：Phase 1 完成后必须通过 §5.6 的 COM 初始化检查清单逐项确认。

### 16.6 Qt 兼容性退化（新增）

风险：替换 plugin_manager.py 的 Qt 剪切板为 Win32 剪切板后，Qt 的 `dataChanged` 信号不再触发，依赖此信号的插件/内置功能中断。

应对：

- ClipboardService 内置 QtClipboardBridge：感知 Win32 剪切板变化后手动发射 `QClipboard.dataChanged` 信号。
- Phase 1-3 保留 PluginAPI 的 Qt 剪切板实现作为兼容路径。
- Phase 4 统一后再移除 Qt 兼容层。

### 16.7 性能退化（新增）

风险：统一服务层引入额外抽象开销，弹窗打开延迟从 <50ms 增加到 >200ms。

应对：

- 为弹窗打开路径建立性能基线（p50 < 30ms, p99 < 50ms）。
- `ClipboardService.read_text()` 区分 fast 模式（无重试、无格式枚举、<1ms）和 full 模式（完整快照）。
- 弹窗打开时只使用 fast 模式，full 模式异步执行。
- 预设量化目标并加入 CI 性能测试。

### 16.8 二进制格式恢复失败（新增）

风险：ClipboardSnapshot 保存富文本/图片/自定义格式的二进制数据，但 SetClipboardData 对某些格式的 HGLOBAL 要求特定内存对齐。

应对：

- Phase 1 只保存和恢复文本、文件列表、HTML 等序列化友好的格式。
- CF_DIB/CF_BITMAP 等图片格式在 Phase 1 只检测不保存二进制内容。
- 恢复失败时记录具体失败的格式 ID，便于诊断。
- 复杂格式恢复失败时至少不写空（保留原剪切板不变）。

### 16.9 回滚策略（新增）

风险：每个 Phase 上线后如发现问题，需要快速回滚到之前的状态。

应对策略：

**Phase 0 — 测试基线**：
- Phase 0 本身不修改业务代码，只需回滚测试文件
- 提交前确认 `git stash` / 分支可快速重置

**Phase 1 — ClipboardService**：
- 保留旧函数实现（如 `read_clipboard_text_old()`）至少 2 个小版本
- 旧文件在新文件无法加载时通过 `try: from clipboard_service import ... except ImportError: pass` 兜底
- 回滚方案：删除 `clipboard_service.py`，恢复旧 `command_variables.py` 的原 `read_clipboard_text` 实现（通过 git revert）

**Phase 2 — SelectedTextService**：
- `ShortcutExecutor._capture_selected_text` 保留为独立静态方法
- `SelectedTextService` 不可用时，`ShortcutExecutor._resolve_command_variables()` 回退到旧 `_capture_selected_text`
- 回滚方案：删除 `selected_text_service.py`，恢复旧代码中的 `_capture_selected_text` 直接调用

**Phase 3+ — 分类器/InteractionContext**：
- 新字段（`clipboard_kind`、`selected_text` 等）默认空值 → 旧消费者不受影响
- `ClipboardClassification` 失败时返回 `kind="unknown"`, `confidence=0.0`
- `InteractionContext` 构建失败时各个子字段返回各自 "空" 值（空字符串/空列表/None）

**通用回滚检查清单（所有 Phase）**：
1. 确保旧 API 在新代码提交前已有兼容包装
2. 确保 `CommandContext` 新字段默认空值
3. 确保所有新增模块不可用时应用仍能正常工作
4. 每个 Phase 完成后单独提交，避免混合提交导致回滚困难
5. 在 changelog 中明确记录每个 Phase 的移除废弃时间表

## 17. 建议优先级

推荐实施顺序（P0→P1→P2→P3→P4 分级）：

| 优先级 | Phase | 内容 | 预计工时 | 前置依赖 |
|--------|-------|------|---------|---------|
| P0 | 0 | 基线测试锁定（含 COM 初始化测试 + 双路径一致性测试） | 3-4天 | 无 |
| P0 | 1 | ClipboardService（读写+快照+重试+COM STA+QtBridge+shortcut_url_exec替换） | 6-8天 | Phase 0 |
| P0 | 2 | SelectedTextService（仅 Clipboard fallback 服务化） | 3-5天 | Phase 1 |
| P1 | 3 | 剪切板分类器（URL/JSON/JWT/颜色/IP/路径） | 3-4天 | Phase 1 |
| P1 | 4 | InteractionContext + 替换剩余旧入口 + 插件 API 扩展 | 5-7天 | Phase 1,2,3 |
| P2 | 5 | 上下文动作推荐 | 5-7天 | Phase 3,4 |
| P2 | 6 | Win32 Edit/RichEdit 非侵入式策略（含 AttachThreadInput 超时） | 4-6天 | Phase 2 |
| **P2→P3** | **7** | **剪切板历史（含隐私过滤 + SQLite 迁移策略）** | **10-14天** | **Phase 1** |
| P4 | 8 | UI Automation 策略（需要外部依赖评估） | 7-10天 | Phase 6 |
| P4 | 9 | 特殊应用适配（终端/Office/浏览器） | 持续 | Phase 6 |

原因：

- **Phase 1 范围扩大**：增加了 `shortcut_url_exec.py` 替换（3 行代码）、COM 初始化策略（核心阻塞依赖）、QtClipboardBridge。预计工时从 5-7 天增加到 6-8 天。
- **Phase 0 工时调整**：增加了 COM 初始化和双路径一致性等高优先级测试用例，从 2-3 天调整到 3-4 天。
- **Phase 7 从 P3 提升到 P2→P3**：隐私过滤规则（§11.4）应该在历史存储之前就设计好；AddClipboardFormatListener 在 Phase 1 之后就能接入。但 UI 和存储成本不变，所以标记为过渡优先级。
- **Phase 1→2→3→4** 是核心架构改造，P0/P1 必须优先完成。
- **Phase 2（SelectedTextService）应在 Phase 3（分类）之前**，因为选中文字是 `{{selected_text}}` 的阻塞依赖。
- **分类（Phase 3）和 InteractionContext（Phase 4）** 为动作推荐打基础。
- **非侵入式策略（Phase 6）** 不阻塞架构改造，但能减少剪切板污染，P2 合理。Phase 6 的关键风险（AttachThreadInput 死锁）已经在 §6.3 中解决。
- **历史功能（Phase 7）** 价值高但隐私和 UI 成本更高，调整为 P2→P3。
- **UI Automation（Phase 8）** 需要引入外部依赖（comtypes/uiautomation），放在最后评估。如果 Phase 6 的 Win32 Edit 策略覆盖了 60% 以上的常见场景，可以考虑不做 UIA。

## 19. 关键验收标准

### 19.1 功能标准

- 剪切板读写只有一个核心服务入口。
- 选中文字读取只有一个核心服务入口。
- `{{selected_text}}` 不再直接依赖 `ShortcutExecutor` 内部私有逻辑。
- 插件可以通过权限化 API 读取剪切板和选中文字。
- 弹窗能识别当前剪切板类型并推荐动作。
- 剪切板历史可开关、可搜索、可清空。
- 选中文字读取失败有可诊断原因（返回 error 字段而非静默空字符串）。
- 所有外部输入进入 shell 命令前继续强制安全引用。
- 不把剪切板全文写入日志。
- 现有剪切板变量、内置命令、插件、选中文件能力不回归。

### 19.2 量化标准

| 指标 | 目标 | 测量方法 |

| 弹窗打开延迟（p99） | ≤ 50ms | 同上，取 99 百分位 |
| 剪切板读取成功率 | ≥ 99%（重试机制下） | CI 中模拟 `OpenClipboard` 被占用 5 次 → 第 6 次成功；运行 200 次 |

| 选中文字读取成功率（VS Code） | ≥ 90%（Clipboard fallback 策略） | 同上 |

| 新增测试用例数 | ≥ 45 个 | 按 §14 测试用例表计数 |
