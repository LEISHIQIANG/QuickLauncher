# QuickLauncher 开机启动优化方案

## 问题分析

### 当前实现
- 使用注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`
- 启动时机：用户登录 → 桌面加载完成 → 执行 Run 键程序
- 启动延迟：通常 10-30 秒

### 为什么慢？

1. **启动时机晚**
   - Run 键优先级最低，在所有系统组件加载完才执行
   - 火绒/Clash 使用服务或任务计划程序，登录时立即启动

2. **Python 启动开销**
   - 即使打包成 exe，仍需解压运行时（50-100MB）
   - 加载大量依赖：PyQt6、PIL、pywin32、pynput 等
   - 火绒/Clash 是原生 C++ 编写，启动只需几毫秒

3. **初始化流程重**
   - 清理日志、读配置、检查单实例
   - 创建 QApplication、托盘、IPC 服务器
   - 同步加载，阻塞启动

## 优化方案

### 方案 1：任务计划程序（推荐）⭐

**效果**：提前 5-10 秒启动

**原理**：
- 任务计划程序在用户登录时立即触发，不等桌面加载
- 可设置 3 秒延迟，避免开机卡顿

**实现**：
```python
# 已创建 core/task_scheduler_manager.py
from core.task_scheduler_manager import enable_task_scheduler

# 启用（延迟 3 秒）
enable_task_scheduler(delay_seconds=3)
```

**优点**：
- 无需修改主程序
- 用户可在"任务计划程序"中管理
- 兼容性好

**缺点**：
- 仍有 Python 启动开销
- 需要用户授权创建任务

---

### 方案 2：延迟初始化（最简单）⭐⭐

**效果**：感知启动时间减少 50%

**原理**：
- 先显示托盘图标（用户看到程序已启动）
- 后台异步加载配置窗口等

**实现**：
修改 main.py，将非核心模块启动改为异步：

```python
# 启动托盘应用
_tray_app = TrayApp()
logger.info("托盘应用启动成功")

# 立即进入事件循环
app.exec()
```

**优点**：
- 改动最小
- 立即见效
- 用户体验提升明显

---

### 方案 3：启动器 + 主程序分离（最快）⭐⭐⭐

**效果**：启动时间 < 1 秒（接近火绒/Clash）

**原理**：
- 用 C++/Rust 写一个 5KB 的启动器
- 启动器只做：显示托盘图标 → 后台启动 Python 主程序
- 用户感知：秒启动

**实现**：
```cpp
// launcher.cpp (编译成 QuickLauncherStarter.exe)
#include <windows.h>

int WINAPI WinMain(HINSTANCE hInstance, HINSTANCE, LPSTR, int) {
    // 1. 显示托盘图标（原生 Win32）
    NOTIFYICONDATA nid = {sizeof(nid)};
    nid.hIcon = LoadIcon(NULL, IDI_APPLICATION);
    Shell_NotifyIcon(NIM_ADD, &nid);

    // 2. 后台启动主程序
    ShellExecute(NULL, "open", "QuickLauncher.exe", NULL, NULL, SW_HIDE);

    // 3. 等待主程序接管托盘后退出
    Sleep(3000);
    Shell_NotifyIcon(NIM_DELETE, &nid);
    return 0;
}
```

**优点**：
- 启动速度接近原生程序
- 用户体验最佳

**缺点**：
- 需要额外开发 C++ 启动器
- 打包流程变复杂

---

### 方案 4：Windows 服务（不推荐）

**效果**：开机即启动（最快）

**缺点**：
- 需要管理员权限安装
- 服务无法显示 GUI（需要额外通信）
- 用户卸载困难
- 杀毒软件可能拦截

---

## 推荐实施步骤

### 第一步：延迟初始化（立即实施）
修改 main.py，异步加载非核心模块，立即进入事件循环。

### 第二步：任务计划程序（替换注册表）
在设置界面添加选项：
- [ ] 使用注册表启动（兼容模式）
- [x] 使用任务计划程序启动（推荐，更快）

### 第三步（可选）：启动器分离
如果追求极致速度，开发 C++ 启动器。

## 火绒/Clash 的实现

**火绒**：
- Windows 服务 + 原生 C++ 驱动
- 开机即加载内核驱动，用户登录时服务已运行

**Clash**：
- 任务计划程序 + Go 编译（启动快）
- 延迟初始化：先启动核心，UI 后加载

**共同点**：
1. 不用 Python（启动开销小）
2. 最小化启动流程（核心功能优先）
3. 使用更早的启动时机（服务/任务计划程序）
