## 1. 全局低级钩子生命周期死锁与竞态冲突

> [!IMPORTANT]
> **业务痛点**：在打开原生“文件/目录选择对话框”（`QFileDialog`）或模态消息框时，频繁触发“鼠标转圈、界面卡死”或“意外唤醒中键弹出菜单”等系统冲突。

### 1.1 根源剖析
1. **`SetWindowsHookEx` 物理模块句柄传递错误**：
   * 原有代码传入了 `GetModuleHandle(NULL)`。根据 Win32 规范，注册全局低级钩子（`WH_MOUSE_LL`）的第三个参数必须是包含钩子回调函数的 DLL 物理模块句柄。`GetModuleHandle(NULL)` 返回的是主进程可执行文件（`python.exe`）的句柄，导致 Windows 操作系统内部归属错误的 DLL 引用计数，使得 DLL 卸载时物理残留或内存页回退引发崩溃。
2. **卸载重装时的“双线程覆盖与孤儿钩子”竞态**：
   * Python 调用 `uninstall()` 时，C++ `UninstallMouseHook` 向旧线程发送 `WM_QUIT` 退出信号，随后主线程调用 `WaitForSingleObject` 仅等待 **10 毫秒**。
   * Windows 线程时间片通常为 **15.6ms**，10ms 的极短等待导致 `WaitForSingleObject` 超时。主线程超时后强行关闭线程句柄，并立即调用 `InstallMouseHook` 启动**新线程（Thread B）**。
   * Thread B 迅速执行并调用 `SetWindowsHookEx`，将新钩子句柄写入全局变量 `g_mouseHook`。
   * 随后，旧线程（Thread A）被操作系统调度唤醒，执行收尾清理，在临死前顺手把新钩子句柄 `g_mouseHook` 给卸载了！而 Thread A 自己的旧钩子由于句柄被覆盖丢失，永远残留在 OS 钩子链中变成“孤儿钩子”，不受暂停控制地发射中键信号。
3. **C++ 后台日志线程引发的 Windows 跨线程死锁**：
   * 在模态对话框打开期间，主线程正阻塞等待。此时 C++ 后台日志线程 `DebugLogThreadProc` 调用 `GetWindowTextA` 查询当前进程内部的主窗口标题。
   * 当对同一进程但不同线程的窗口句柄调用 `GetWindowTextA` 时，OS 会在内部向该窗口所有者线程发送一条同步的 `WM_GETTEXT` 消息。由于主线程正因为 modal 对话框同步等待而无法响应消息，日志线程会被**永久死锁**。
4. **模态焦点切换过渡期的微秒级队列竞态**：
   * 模态窗弹出瞬间，Windows 底层输入队列可能会遗留未被派发的键鼠消息，导致在 Python 已设置 `paused = True` 后，底层残留的消息依然穿透 C++ 拦截，送达 Python 呼起二级弹窗。

### 1.2 修复方法
1. **模块句柄严谨动态提取**：
   新增 `GetCurrentDllModule()` 辅助函数，基于函数本身的物理指针，动态获取 DLL 真实的 `HMODULE`：
   ```cpp
   static HMODULE GetCurrentDllModule() {
       HMODULE hMod = NULL;
       GetModuleHandleExA(
           GET_MODULE_HANDLE_EX_FLAG_FROM_ADDRESS | GET_MODULE_HANDLE_EX_FLAG_UNCHANGED_REFCOUNT,
           reinterpret_cast<LPCSTR>(&GetCurrentDllModule),
           &hMod
       );
       return hMod;
   }
   ```
2. **引入线程自愈安全屏障（Thread-Alive Sentinel）**：
   引入了 `g_mouseThreadAlive` / `g_keyboardThreadAlive` 原子变量，在 `Install` 函数中部署**自旋防撞车屏障**，阻断新钩子线程的初始化，直到旧线程完全安全退出，并将 join 超时由 10ms 提升至安全的 **500ms**：
   ```cpp
   if (g_mouseThreadAlive.load()) {
       int waitCount = 0;
       while (g_mouseThreadAlive.load() && waitCount < 50) {
           std::this_thread::sleep_for(std::chrono::milliseconds(10));
           waitCount++;
       }
   }
   ```
3. **同进程窗口 GetWindowTextA 死锁豁免审计**：
   在 `WindowSummary` 增加 PID 审计，对同进程窗口**直接豁免 `GetWindowTextA` 跨线程调用**：
   ```cpp
   if (pid != GetCurrentProcessId()) {
       GetWindowTextA(hwnd, title, sizeof(title));
   } else {
       lstrcpynA(title, "current_process_window", sizeof(title));
   }
   ```
4. **Python-level 终极避让过滤**：
   在 Python 的事件接收首部，新增了逻辑层的**防穿透终极滤网**。若处于 paused 状态，100% 拒绝任何信号分发，不抛出任何 Qt 信号：
   ```python
   paused_state = False
   if hasattr(self, "mouse_hook") and self.mouse_hook:
       try:
           paused_state = self.mouse_hook.is_paused()
       except Exception:
           pass
   if paused_state is True:
       logger.info("HOOK_CALLBACK_IGNORED hook_paused pos=(%s,%s)", x, y)
       return
   ```

## 2. 数据管理器（DataManager）线程安全与原子配置迁移

> [!IMPORTANT]
> **业务痛点**：多线程或后台任务读取数据时可能出现数据竞争，且配置由老旧目录迁移至新目录时非原子操作，中途断电或崩溃会导致配置文件损坏或丢失。

### 2.1 根源剖析
1. **数据读取路径无保护**：
   * `get_settings()`、`backup_full_config()` 等读路径返回了原始数据的活引用，后台线程对数据的操作极易引起主线程的 `RuntimeError: dictionary changed size during iteration` 或数据脏读。
2. **数据迁移无回滚和前向恢复机制**：
   * 原有配置迁移在迁移失败或中途退出时，可能导致老配置被删而新配置尚未写入，造成灾难性的用户数据丢失。

### 2.2 修复方法
1. **全路径读写保护与深拷贝隔离**：
   * 在 `DataManager` 内部，对所有读写接口进行加锁保护。`get_settings()` 统一加锁并返回 **`copy.deepcopy(settings)`**，避免活引用暴露。
   * 对 `backup_full_config()`、`export_shareable_config()` 等外部数据交互接口，统一接入互斥锁控制。
2. **建立原子级配置迁移与前向自愈系统**：
   * **备份机制**：配置迁移前自动创建完整的本地备份至 `_BACKUP_DIR`。
   * **原子标志位**：引入 `.migration_done` 原子标志文件，并在启动初始化时调用 `needs_partial_recovery()` 进行诊断。如果新配置迁移中断导致新目录配置文件丢失而 marker 不存在，系统会自动启动前向恢复流程 `recover_partial()`，无缝拉起最近一份正常备份。

## 3. 核心层（Core）反向导入 UI 架构解耦

> [!IMPORTANT]
> **业务痛点**：`core/interaction_context.py`、`core/plugin_manager.py` 和 `core/shortcut_command_exec.py` 等底层模块中直接导入了 `ui/` 包的 UI 组件或函数，导致产生了复杂的循环导入隐患，削弱了测试的独立性。

### 3.1 根源剖析
* 核心逻辑模块为了在完成命令执行或插件状态切换时同步显示特定的 Toast 提示或弹框界面，直接使用了物理级 UI 类（例如从 `ui.launcher_popup.window_detection` 或 `ui.toast` 导入方法），强行将界面渲染绑定在核心调度模块中。

### 3.2 修复方法
1. **抽象回调与依赖注入（Dependency Injection）**：
   * 剥离 core 模块中所有直连的 UI 导入。
   * 引入统一的 `InteractionContext` 交互上下文，利用回调委托（Delegate）或观察者模式，在程序初始化时将 UI 的弹框及 Toast 方法注册进 Core。Core 执行逻辑时仅调用抽象的回调函数，不再关心具体的 UI 类与渲染机制。
2. **包边界防御规则**：
   * 移除了所有的反向依赖。对于极少数必须和 UI 进行交互的场景，封装了专用的底层事件接口，保证 `core` 是完全独立于 UI 框架的可测单元。

## 4. 快捷方式健康检查与修复预览确认机制

> [!WARNING]
> **业务痛点**：快捷方式健康扫描（`shortcut_health_window.py`）在扫描出失效或受损配置时，直接一键静默自动执行“暴力删除或修复”，使用户对数据变更完全处于不知情状态。

### 4.1 根源剖析
* 历史代码只提供了一键式执行体，未进行“只读扫描/修复预览”的隔离，也没有引入用户确认的第二道阀门，极易误删正常但暂时由于外部环境被标记为不可访问的快捷路径。

### 4.2 修复方法
1. **只读扫描与预览机制分离**：
   * 新增了只读模式的 `preview_health_fixes()` 接口，只计算修复清单，绝不写入磁盘。
2. **引入分组别确认 UI**：
   * UI 层对安全修复项（如图标损坏）和高危删除项（如失效快捷方式物理删除）进行颜色分组，强关联“确认提示窗”。
   * 必须在用户点击“应用已选修复”并二次弹窗明确授权后，方可批量执行修改写入。

## 5. 静默异常拦截门禁与 Pre-commit 规则引入

> [!WARNING]
> **业务痛点**：代码中历史遗留了高达 **468 处** 毫无日志和解释的静默异常处理（`except Exception: pass`），导致运行期的错误完全不可诊断。

### 5.1 根源剖析
* 开发人员习惯于用 `try-except pass` 规避界面闪退或处理未定义的系统异常，导致底层报错被完全掩盖，许多底层崩溃转化为难以被日志捕获的幽灵缺陷。

### 5.2 修复方法
1. **Pre-commit 强力检查门禁**：
   * 新增了自动化静态扫描脚本 `scripts/check_silent_exceptions.py`。
   * 在 [`.pre-commit-config.yaml`](file:///G:/LEI-PLUG/QuickLauncher/QuickLauncher_V1.6.1.0/.pre-commit-config.yaml) 中挂载为强制性的 Git Commit Hook：
   ```yaml
   - repo: local
     hooks:
       - id: no-silent-exceptions
         name: Prevent new silent exceptions
         entry: python scripts/check_silent_exceptions.py
         language: python
         files: ^(core|ui|services|hooks)/.*\.py$
   ```
2. **规范化异常哨兵**：
   * 任何新提交的 `except Exception: pass` 均会被 pre-commit 直接拦截并拒绝提交。
   * 推广统一的 `@safe_method` 异常处理器与 `safe_execute()`，强制在底层捕获时输出至少 `logger.debug` 级别的诊断线索或 `exc_info=True` 调用栈。

## 6. 新手引导动态版本化与安全模式文档同步

> [!NOTE]
> **业务痛点**：`welcome_guide.py` 内部硬编码了历史过期版本号，与当前主程序的真实版本无法对齐，且安全模式（`--safe-mode`）缺乏用户能查阅的手册，难以推广。

### 6.1 根源剖析
* 迎新导航和主版本机制没有打通，在补丁包推送或重新打包时，迎新页面显示滞后。

### 6.2 修复方法
1. **新手引导动态绑定**：
   * 将迎新版本字段移接至 `core.version.APP_VERSION`，实现程序版本的一键更新，动态变更向导卡片。
   * 优化了“稍后提醒”、“不再显示”的配置存储机制。
2. **安全模式完整文档化**：
   * 在 [README.md](file:///G:/LEI-PLUG/QuickLauncher/QuickLauncher_V1.6.1.0/README.md) 中完整归档了安全模式的技术原理与运维指南，详细描述了 `--safe-mode` 对钩子绕过、独立数据保护的逻辑，便于用户在遭遇系统死锁时进行一键式降级排障。
