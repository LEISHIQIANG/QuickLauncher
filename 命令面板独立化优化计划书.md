# QuickLauncher 命令面板独立化优化计划书

## 1. 最终产品决策

中键弹窗以后只作为快捷入口和命令入口，不承载命令结果显示功能。用户点击图标或命令后，中键弹窗应立即消失。凡是需要展示结果、进度、参数输入、确认、历史或后续操作的命令，独立命令面板负责显示。

目标交互：

```text
中键呼出 QuickLauncher
  -> 点击命令图标或执行 slash 命令
  -> 中键弹窗立即隐藏
  -> 如命令需要展示结果/输入/进度，独立命令面板显示
  -> 如命令只是打开窗口/执行动作，无需结果面板
```

这条原则比“中键弹窗显示摘要，复杂结果再展开”更干净：启动器保持快进快出，命令面板负责阅读、复制、保存、重试、查看历史和继续操作。

## 2. 当前代码检查结论

### 2.1 中键弹窗当前承担了结果显示

当前结果显示主要在：

```text
ui/launcher_popup/popup_command_result.py
```

它会在启动器图标网格上覆盖结果卡片，创建 `QTextEdit` 显示文本，手绘 action、表格、二维码和进度条，并临时 pin 住中键弹窗。这与“点击后立即消失”的新交互冲突。

### 2.2 执行入口明确

点击图标执行逻辑在：

```text
ui/launcher_popup/popup_item_execution.py
```

其中 `_execute_item()` 负责运行时输入、panel 命令特殊处理、后台线程执行和错误显示。新方案的关键改造点就在这里：不要再调用 `show_command_result()`，而是隐藏弹窗并把需要交互的命令交给独立命令面板。

### 2.3 独立窗口基础已经存在

项目已有：

```text
ui/themed_tool_window.py
```

它提供无边框工具窗口、标题栏、主题跟随、亚克力/模糊、拖动、文本框/列表/按钮样式。新命令面板应复用它，保持与 `ui/log_window.py`、`ui/slash_help_window.py`、`ui/diagnostics_window.py` 等窗口一致。

### 2.4 命令协议已有基础

`core/command_registry.py` 已有 `CommandResult`、`CommandAction`、`CommandContext`、`CommandDefinition`。独立命令面板应直接消费这些模型，不新增另一套结果协议。

当前结果传递依赖：

```python
set_pending_command_result(result)
take_pending_command_result()
```

这是全局单槽，只适合迁移期桥接，不适合命令历史、异步进度和独立面板。需要新增正式的结果存储层。

### 2.5 迁移期开关需要清理

`ui/launcher_popup/popup_item_execution.py` 中读取 `settings.enable_command_panel_v2`，但 `AppSettings` 当前没有这个字段。新方案应把独立命令面板作为正式能力，不再依赖这个迁移期开关。

## 3. 新架构

建议拆成四层：

```text
LauncherPopup
  只负责入口，点击后隐藏

CommandPanelWindow
  负责参数输入、执行中状态、进度、结果、历史、action

CommandResultStore
  保存最近命令结果和动作链报告

CommandExecutionService
  统一执行 registry 命令、普通 CMD/Python 捕获命令、插件命令
```

## 4. 新增模块

### 4.1 core/command_results.py

内存保存最近结果。第一版只保留最近 50 条，不落盘。

```python
@dataclass
class StoredCommandResult:
    id: str
    command_id: str = ""
    command_title: str = ""
    raw_input: str = ""
    source: str = ""
    created_at: float = 0.0
    duration: float = 0.0
    result: CommandResult = field(default_factory=CommandResult)
```

```python
class CommandResultStore:
    def add(self, result: CommandResult, **meta) -> str: ...
    def get(self, result_id: str) -> StoredCommandResult | None: ...
    def latest(self) -> StoredCommandResult | None: ...
    def list(self) -> list[StoredCommandResult]: ...
    def clear(self) -> None: ...
```

### 4.2 ui/command_panel_window.py

独立命令面板窗口，继承 `ThemedToolWindow`。

第一版布局：

```text
标题栏：命令面板
副标题：命令名称 / 状态 / 时间

顶部输入区：
  命令或参数输入
  [执行] [取消]

内容区：
  text/log/table/kv/list/qr/progress

底部：
  [复制] [保存] [重新执行] [更多] [关闭]
```

MVP 可以先只支持 text/log，但类设计要预留 table/kv/list/qr/progress。

### 4.3 core/command_execution_service.py

建议后续新增统一执行服务，逐步收拢当前分散在 `popup_item_execution.py` 和 `shortcut_command_exec.py` 中的命令处理。

职责：

- 解析 slash 命令。
- 构建 `CommandContext`。
- 调用 `CommandDefinition.handler`。
- 处理 CMD/Python 捕获输出。
- 写入 `CommandResultStore`。
- 通知 `CommandPanelWindow` 更新状态。

## 5. LauncherPopup 改造规则

### 5.1 点击后立即隐藏

panel 命令不再执行：

```python
self.show()
self.raise_()
self.activateWindow()
self.show_command_result(...)
```

新路径：

```python
self.hide()
self.tray_app.show_command_panel(command_id, args_text, raw_input)
```

如果没有 `tray_app`，可以 fallback 到临时窗口，但常规路径必须由 TrayApp 持有窗口引用，避免生命周期问题。

### 5.2 运行时输入迁移

当前 `{input}` 在弹窗层用 `ThemedInputDialog` 收集。未来应迁移到命令面板：

```text
点击命令
  -> 中键弹窗隐藏
  -> 命令面板打开
  -> 面板显示输入项
  -> 用户确认后执行
```

这样输入不会阻塞启动器。

### 5.3 不需要面板的命令

不需要显示命令面板：

- 打开配置窗口。
- 打开文件夹。
- 打开 URL。
- 置顶/取消置顶。
- 普通程序启动。
- 不捕获输出的静默命令。

需要显示命令面板：

- `CommandDefinition.interaction_mode == COMMAND_INTERACTION_PANEL`。
- 命令需要参数输入。
- 命令返回需要阅读的 `CommandResult`。
- 普通 CMD/Python 开启捕获输出。
- 动作链执行需要步骤报告。
- 异步任务需要进度或取消。

## 6. TrayApp 改造

在 TrayApp 或窗口 mixin 中维护：

```python
self.command_panel_window = None
self.command_result_store = CommandResultStore()
```

新增入口：

```python
def show_command_panel(self, command_id="", args_text="", raw_input="", result_id=None):
    ...
```

建议放在：

```text
ui/tray_mixins/windows_mixin.py
```

因为日志窗口、slash 帮助、诊断窗口都由这里管理。

## 7. CommandResult 展示策略

命令面板完整支持：

```text
text
log
table
kv
list
progress
qr
confirm
```

中键弹窗不再负责这些展示类型。`ui/launcher_popup/popup_command_result.py` 可保留兼容期，但新增 display type 的完整渲染应放到 `CommandPanelWindow`。

## 8. 与动作链的关系

动作链点击后：

```text
中键弹窗隐藏
动作链后台执行
命令面板显示执行状态
执行完成后命令面板显示每步结果
```

动作链结果使用：

```python
CommandResult(
    display_type="list",
    payload={"items": [...]}
)
```

每步记录：

```text
name
kind
success
skipped
duration
error
output_summary
```

## 9. 与命令增强的关系

命令增强计划中的以下能力都应以独立命令面板为承载：

- CMD/Python 捕获输出。
- table/kv/list/log 渲染。
- 更多 action。
- 参数化命令输入。
- 异步进度和取消。
- 命令历史。

## 12. 验收标准

第一轮完成后：

- 中键弹窗不再显示命令结果。
- 点击需要命令面板的命令后，中键弹窗立即消失。
- 独立命令面板显示结果、错误和主要 action。
- 短命令、长命令、插件命令都走同一命令面板。
- 命令面板风格与现有工具窗口一致。
- 中键弹窗启动体验不被命令结果打断。

## 13. 2026-05-26 修订：窗口尺寸模板与 UI 约束

独立命令面板需要提供固定的窗口尺寸模板，由命令或插件内部代码决定使用哪一种。除自动调整模板外，用户不能手动调整窗口大小。

### 13.1 四种窗口尺寸模板

建议在 `ui/command_panel_window.py` 中定义统一模板：

```python
COMMAND_PANEL_SIZE_PRESETS = {
    "small": (360, 420),
    "medium": (520, 606),
    "large": (720, 840),
    "auto": None,
}
```

具体像素可以在实现时根据当前中键弹窗实际尺寸、DPI 和视觉效果微调，但要求：

- `small` 尺寸接近当前中键弹窗。
- `medium` 比 small 明显更宽高，适合普通命令报告。
- `large` 再大一些，适合日志、表格、动作链报告。
- `auto` 根据内容动态调整。
- 除 `auto` 外，`small/medium/large` 的长宽比例尽可能一致。
- 除 `auto` 外，窗口不可由用户手动调整大小。

如果当前中键弹窗尺寸不是固定值，`small` 可按运行时弹窗计算值或默认配置生成：

```text
small.width  ≈ LauncherPopup.width
small.height ≈ LauncherPopup.height
```

但模板最终应落为稳定尺寸，避免不同命令打开时窗口忽大忽小。

### 13.2 命令和插件如何选择尺寸

命令或插件可以通过 `CommandResult.payload` 指定尺寸：

```python
CommandResult(
    display_type="table",
    payload={
        "window_size": "medium",
        "columns": [...],
        "rows": [...],
    },
)
```

也可以在命令定义层增加默认值：

```python
CommandDefinition(
    ...,
    result_window_size="small",
)
```

如果两者都存在，优先级：

```text
CommandResult.payload.window_size
CommandDefinition.result_window_size
display_type 默认规则
全局默认 medium
```

默认规则建议：

```text
text      small
kv        small 或 medium
list      medium
table     large
log       large
progress  small
qr        small
动作链     large
```

### 13.3 自动调整模板

`auto` 模板用于内容长度差异很大的命令。实现要求：

- 根据内容计算推荐宽高。
- 设置最小尺寸和最大尺寸。
- 最大尺寸不能超过当前屏幕可用区域的 80%。
- 允许用户调整大小。
- 记住同类命令本次打开期间的调整，不必第一版持久化。

建议范围：

```text
min: small
max: 当前屏幕可用区域 80%
```

### 13.4 固定模板不可调整

`small/medium/large` 模板应调用：

```python
self.setFixedSize(width, height)
```

`auto` 模板才允许：

```python
self.resize(width, height)
self.setMinimumSize(...)
self.setMaximumSize(...)
```

这样可以保证命令面板体验可预测，由命令或插件按结果类型选择窗口大小，而不是让用户每次手动调整。

### 13.5 UI 风格复用

命令面板 UI 必须参考并复用现有项目样式：

- 窗口壳复用 `ThemedToolWindow`。
- 按钮样式复用 `ThemedToolWindow.style_buttons()`。
- 文本结果复用 `style_plain_text()`。
- 列表结果复用 `style_list_widget()` 或 `style_compact_list_widget()`。
- 滚动条复用 `StyleSheet.get_scrollbar_style(theme)`。
- 颜色、透明、边框、圆角跟随现有 `LogWindow`、`SlashHelpWindow`、`DiagnosticsWindow` 风格。

不要为命令面板引入新的大面积视觉语言。它应该看起来像 QuickLauncher 已有工具窗口的一部分。

### 13.6 MVP 尺寸验收

第一版需要至少验证：

- `small` 打开后和当前中键弹窗体量接近。
- `medium`、`large` 与 `small` 保持近似比例。
- `small/medium/large` 不能拖拽调整大小。
- `auto` 可以根据内容变大，并可调整。
- 插件返回 `payload.window_size` 后能正确选择模板。

## 14. 三条优化路径的实施顺序

三条路径的依赖关系应按基础设施优先，而不是按功能吸引力优先。

推荐顺序：

```text
1. 命令面板独立化
2. 命令增强
3. 动作链
```

原因：

- 命令增强的捕获输出、结构化结果、参数输入、异步进度都需要一个稳定的显示承载。如果先做命令增强，又继续显示在中键弹窗里，后面迁移到独立面板会返工。
- 动作链的执行过程天然需要状态、步骤结果、错误详情和重新执行入口。如果没有独立命令面板，动作链结果只能临时塞进弹窗或日志，体验和代码都会走偏。
- 动作链中的命令步骤会复用命令增强能力，例如捕获输出、`CommandResult`、`CommandAction`、窗口尺寸模板。因此命令增强协议应先稳定。

### 14.1 第一阶段：命令面板基础设施

先完成：

```text
core/command_results.py
ui/command_panel_window.py
ui/tray_mixins/windows_mixin.py 的 show_command_panel()
ui/launcher_popup/popup_item_execution.py 的 panel 命令隐藏弹窗并转交命令面板
```

必须稳定的接口：

```python
CommandResultStore.add(result, **meta) -> result_id
TrayApp.show_command_panel(command_id="", args_text="", raw_input="", result_id=None)
CommandPanelWindow.show_result(result_id)
CommandPanelWindow.run_command(command_id, args_text, raw_input)
```

这一阶段只需要支持 `text/log`，但必须把窗口尺寸模板做好，因为后续命令和插件会直接依赖 `payload.window_size`。

### 14.2 第二阶段：命令增强接入命令面板

在命令面板基础稳定后，再做：

```text
CMD/Python 捕获输出
CommandResult display_type: table/kv/list/progress/qr
CommandAction 完整操作区
CommandDefinition.params 参数输入
异步任务进度和取消
```

这一阶段的重点是让所有命令结果进入同一个面板体系，不再扩展 `popup_command_result.py`。

### 14.3 第三阶段：动作链核心与编辑器

动作链应在命令面板和命令结果协议稳定后启动：

```text
ShortcutType.CHAIN
core/shortcut_chain_exec.py
ChainDialog 左右两栏布局
图标栏空白处右键入口
动作链执行结果写入 CommandResultStore
CommandPanelWindow 显示步骤结果
```

动作链第一版可以只引用已有快捷方式，不做复杂变量和条件。它的完整结果依赖命令面板 `list` 渲染和 `large/auto` 窗口模板。

### 14.4 可并行但要避免冲突的工作

可以并行：

- 命令面板 UI 外壳和 `CommandResultStore`。
- 命令增强协议文档和插件模板更新。
- 动作链数据模型草案和测试设计。

不建议并行：

- 在 `popup_command_result.py` 继续扩复杂结果 UI。
- 在动作链里自建一套结果窗口。
- 在插件里各自实现不同窗口尺寸和展示样式。

### 14.5 最小闭环

最小可交付闭环应是：

```text
中键弹窗点击 /uuid
  -> 弹窗立即隐藏
  -> 独立命令面板 small 模板打开
  -> 显示 UUID
  -> 可复制
```

第二个闭环：

```text
中键弹窗点击捕获输出命令
  -> 弹窗立即隐藏
  -> 独立命令面板 large 或 auto 模板打开
  -> 显示 stdout/stderr
  -> 可保存输出
```

第三个闭环：

```text
配置窗口图标栏空白处右键新建动作链
  -> 左右两栏 ChainDialog 编辑步骤
  -> 执行动作链
  -> 弹窗隐藏
  -> 命令面板显示每步结果
```
